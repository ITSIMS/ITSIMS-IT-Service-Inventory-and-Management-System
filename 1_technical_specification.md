## 1. Цель документа

Данный документ определяет требования заказчика к **Автоматизированной информационной системе учёта IT-сервисов (ITSIMS)** на уровне бизнес-задач и функциональности системы.

**Техническое задание (ТЗ)** является исходным документом для разработки детализированных системных требований и последующей технической реализации системы. Документ отвечает на вопрос "**ЧТО** должна делать система" с точки зрения бизнеса, не вдаваясь в технические детали реализации.

**Целевая аудитория:** Заказчик, бизнес-аналитики, менеджеры проекта, системные архитекторы.

## 2. Термины, определения и сокращения

### 2.1. Базовые термины

*   **Заказчик** — Юридическое лицо (организация), инициирующее внедрение Системы с целью централизации управления IT-сервисами в своей организации
*   **Система** — Автоматизированная информационная система учёта IT-сервисов (ITSIMS)
*   **Пользователь** — Сотрудник организации-заказчика, имеющий доступ к Системе для выполнения своих рабочих функций
*   **Администратор системы** — Пользователь с расширенными правами для управления Системой, учётными записями и настройками

### 2.2. Специфичные термины

*   **IT-сервис (Сервис)** — Совокупность IT-активов (ПО, оборудование, данные), предоставляющая определенную ценность пользователям и поддерживающая бизнес-процессы
*   **Досье сервиса** — Карточка сервиса в Системе, агрегирующая всю информацию о нем: метаданные, связи, файлы, историю изменений
*   **Графовая модель** — Способ представления данных, где сервисы являются узлами (вершинами), а зависимости между ними — направленными ребрами (дугами)
*   **Зависимость** — Отношение между сервисами, при котором функционирование одного сервиса (зависимого) требует корректной работы другого (обеспечивающего)

### 2.3. Сокращения и аббревиатуры

*   **API (Application Programming Interface, англ.)** — Программный интерфейс приложения
*   **CMDB (Configuration Management Database, англ.)** — База данных управления конфигурациями
*   **CRUD** — Create, Read, Update, Delete (Создание, Чтение, Обновление, Удаление)
*   **HelpDesk** — Система обработки обращений
*   **PDF (Portable Document Format, англ.)** — Портативный формат документов
*   **RBAC (Role-Based Access Control, англ.)** — Управление доступом на основе ролей
*   **REST (Representational State Transfer, англ.)** — Архитектурный стиль для распределённых систем
*   **SLA (Service Level Agreement, англ.)** — Соглашение об уровне обслуживания
*   **ТЗ** — Техническое задание
*   **XML (eXtensible Markup Language, англ.)** — Расширяемый язык разметки

## 3. Общие сведения

**3.1. Полное наименование Системы:** Автоматизированная информационная система учёта IT-сервисов (далее – Система).

**3.2. Заказчик:** Юридическое лицо (организация), инициирующее внедрение Системы с целью централизации управления IT-ландшафтом.

**3.3. Основание для разработки:** Необходимость оптимизации процессов управления IT-активами и сервисами (ITSM), повышения прозрачности их взаимосвязей, обеспечения актуальности эксплуатационной документации, а также снижение рисков, связанных с внесением изменений в инфраструктуру.

**3.4. Результат работ:** Развёрнутый программный комплекс, функционирующий в инфраструктуре Заказчика, настроенный в соответствии с требованиями настоящего ТЗ, а также пакет сопроводительной документации (руководства администратора и пользователя).

## 4. Назначение и цели создания Системы

**4.1. Назначение Системы**  
Система предназначена для комплексной автоматизации деятельности IT-подразделения в части учёта, каталогизации и управления жизненным циклом IT-сервисов. Ключевой задачей является создание единого источника достоверных данных (Single Source of Truth) о составе, конфигурации, зависимостях и документационном обеспечении IT-услуг.

**4.2. Цели создания**  
*   **Формирование единого реестра сервисов:** Создание структурированной базы данных, содержащей полную информацию обо всех IT-сервисах компании, их характеристиках и владельцах.
*   **Обеспечение прозрачности связей:** Выявление и визуализация зависимостей между компонентами инфраструктуры для анализа влияния изменений (Impact Analysis) и ускорения диагностики инцидентов.
*   **Стандартизация хранения документации:** Внедрение унифицированного подхода к организации файловых структур сервисов, обеспечение версионности и разграничения доступа к технической документации.
*   **Повышение эффективности процессов ITSM:** Интеграция с системами Service Desk и мониторинга для обогащения контекста заявок и инцидентов актуальной информацией о сервисах.
*   **Управление безопасностью и доступом:** Реализация ролевой модели (RBAC) и детального аудита действий для защиты чувствительной информации о конфигурации IT-ландшафта.

## 5. Требования к системе

### 5.1. Функциональные требования

#### 5.1.1. Подсистема учёта и реестра сервисов
Система должна обеспечивать ведение иерархического каталога IT-сервисов.

*   **[F1_ST]:** Система должна обеспечивать создание, чтение, обновление и удаление (CRUD) карточек сервисов (включая возможность архивирования).
*   **[F2_ST]:** Система должна поддерживать атрибутивную модель сервиса, включающую: ID, Наименование, Описание, Статус (Запланирован, В эксплуатации, Выведен), Владелец, SLA, Категория, Критичность.
*   **[F3_ST]:** Система должна обеспечивать возможность массового импорта реестра из внешних источников (CSV, Excel, JSON) и массового обновления атрибутов.

#### 5.1.2. Моделирование связей и зависимостей
Система должна реализовывать графовую модель данных для хранения информации о топологии IT-ландшафта.

*   **[F4_ST]:** Система должна поддерживать типизацию связей (например: "Зависит от", "Включает в себя", "Обменивается данными с").
*   **[F5_ST]:** Система должна обеспечивать строгую направленность связей, четко определяя направление зависимости (Сервис А -> Сервис Б).
*   **[F6_ST]:** Система должна осуществлять валидацию создаваемых связей для предотвращения некорректных состояний (включая контроль циклических зависимостей).

#### 5.1.3. Визуализация графа зависимостей
Система должна предоставлять интерактивный графический интерфейс для работы с топологией сервисов.

*   **[F7_ST]:** Система должна визуализировать узлы и связи с возможностью масштабирования (zoom), перемещения (pan) и перегруппировки узлов.
*   **[F8_ST]:** Система должна обеспечивать контекстную навигацию: при выборе сервиса отображать его "соседей" (сервисы, от которых он зависит, и сервисы, которые зависят от него).
*   **[F9_ST]:** Система должна предоставлять возможность фильтрации графа по типам сервисов, слоям архитектуры или критичности.

#### 5.1.4. Управление файловой структурой и документацией
Каждый сервис в Системе должен являться изолированным контейнером для хранения сопроводительной документации.

*   **[F10_ST]:** Система должна поддерживать хранение иерархической структуры вложенных папок и файлов произвольных форматов внутри карточки сервиса.
*   **[F11_ST]:** Система должна обеспечивать возможность применения предустановленных шаблонов файловой структуры (автоматическое создание папок) при создании нового сервиса.
*   **[F12_ST]:** Система должна реализовывать гранулярное управление правами доступа (ACL) на уровне папок для ограничения доступа к определенному контенту (например, "Пароли") для конкретных ролей.

#### 5.1.5. Поиск и навигация
*   **[F13_ST]:** Система должна обеспечивать полнотекстовый поиск по названиям, описаниям и содержимому текстовых документов.
*   **[F14_ST]:** Система должна поддерживать фасетный поиск для фильтрации списка сервисов по набору атрибутов (статус, владелец, дата обновления).
*   **[F15_ST]:** Система должна автоматически отображать список обратных ссылок (Backlinks), показывая контекстные упоминания сервиса в других карточках.

#### 5.1.6. Журналирование и аудит
Все изменения данных в Системе должны версионироваться.

*   **[F16_ST]:** Система должна вести журнал операций, фиксируя факт изменения атрибутов, связей или файлов с указанием автора, метки времени и старого/нового значения.
*   **[F17_ST]:** Система должна предоставлять интерфейс для просмотра хронологии развития сервиса (истории изменений).

### 5.2. Требования к интеграциям

*   **[I1_ST]:** Система должна предоставлять документированный REST API для всех основных сущностей (Сервисы, Связи) для интеграции с внешними системами (CI/CD, мониторинг).
*   **[I2_ST]:** Система должна использовать форматы JSON/XML для обмена структурами данных.
*   **[I3_ST]:** Система должна поддерживать интеграцию с корпоративными службами каталогов (Active Directory, Keycloak) через SSO/LDAP для централизованной аутентификации.

### 5.3. Нефункциональные требования

*   **[NF1_ST]:** Система должна обеспечивать время отклика интерфейса при навигации по графу (до 1000 узлов) не более 2 секунд.
*   **[NF2_ST]:** Архитектура Системы должна поддерживать горизонтальное масштабирование для обеспечения работоспособности при росте объема данных.
*   **[NF3_ST]:** Система должна функционировать в режиме 24/7 с коэффициентом доступности не ниже 99.5%.

### 5.4. Требования к интерфейсу (UI/UX)

*   **[UI1_ST]:** Система должна предоставлять веб-интерфейс, доступный через современный веб-браузер без установки дополнительных плагинов.
*   **[UI2_ST]:** Интерфейс Системы должен быть адаптивным и корректно отображаться на различных разрешениях экранов (Desktop/Tablet).
*   **[UI3_ST]:** Интерфейс управления связями должен поддерживать Drag-and-Drop операции для создания и редактирования зависимостей.
