## 1. Цель документа

Данный документ определяет системные требования к Автоматизированной информационной системе учёта и управления IT-сервисами (ITSIMS). 

Документ предназначен для системных архитекторов, разработчиков программного обеспечения, бизнес-аналитиков, тестировщиков и всех участников команды, работающих над проектом.

Из документа читатель узнает о поведении системы на системном уровне, её архитектуре, основных компонентах, функциях управления каталогом IT-сервисов и документацией, требованиях к безопасности, производительности и интеграциям.

**Принцип:** Системные требования отвечают на вопрос "**КАК** система будет работать на системном уровне", описывая поведение и характеристики системы, но не углубляясь в программную реализацию.



## 2. Термины, определения и сокращения

### 2.1. Базовые термины

- **Заказчик** — АО «БАРС Груп» в лице представителя Суставова Сергея Александровича

- **Система** — Автоматизированная информационная система учёта и управления IT-сервисами (ITSIMS), разрабатываемая в рамках данного проекта

- **Пользователь** — сотрудник организации-заказчика, использующий систему в рамках своих должностных обязанностей и предоставленных прав доступа

- **Администратор системы** — пользователь с полным контролем над системой, включая управление всеми сервисами, категориями, правами доступа и настройками системы. Роль технического директора автоматически обладает правами администратора системы

- **IT-сервис** — единица учёта в системе, представляющая собой программное обеспечение с определённым набором характеристик (название, описание, версия, категория, статус, привязка к аппаратам) и ассоциированной документацией

- **Администратор сервиса** — пользователь, обладающий полными правами на управление конкретным IT-сервисом: редактирование характеристик, управление документацией, распределение прав доступа к данному сервису

- **Специалист по обслуживанию** — пользователь, обладающий правами на просмотр документации по обслуживанию сервиса и регистрацию фактов обслуживания

- **Обычный сотрудник** — пользователь с правами на просмотр доступных ему IT-сервисов и документации для пользователей (инструкции по установке, настройке, использованию)

- **Категория сервиса** — классификационный признак IT-сервиса, определяемый администратором системы на основе справочника категорий

- **Статус сервиса** — текущее состояние IT-сервиса в жизненном цикле (активный, архивный, на обслуживании, в разработке, на тестировании)

### 2.2. Специфичные термины

- **Веб-приложение** — программная система, доступная пользователям через веб-браузер без необходимости установки дополнительного программного обеспечения на клиентское устройство

- **Клиентская часть** — компонент системы, выполняющийся в веб-браузере пользователя и обеспечивающий пользовательский интерфейс

- **Серверная часть** — компонент системы, выполняющийся на сервере и обеспечивающий обработку запросов, управление данными и бизнес-логику

- **Хранилище данных** — компонент системы, обеспечивающий постоянное хранение структурированных данных о сервисах, пользователях, правах доступа и настройках системы

- **Файловое хранилище** — компонент системы, обеспечивающий хранение файлов документации, ассоциированных с IT-сервисами

- **Файловый каталог сервиса** — структурированное хранилище документации, ассоциированное с конкретным IT-сервисом

- **Шаблон категории** — предустановленная структура папок и документов, автоматически создаваемая при добавлении нового IT-сервиса определённой категории

- **Документ для пользователей** — файл в каталоге сервиса, помеченный как доступный обычным сотрудникам (руководства пользователя, инструкции по установке и настройке)

- **Документ для обслуживания** — файл в каталоге сервиса, помеченный как доступный только специалистам по обслуживанию и администраторам (техническая документация, регламенты обслуживания)

- **Запись обслуживания** — фиксация факта проведения работ по обслуживанию IT-сервиса с указанием даты, описания работ и исполнителей

- **Связь сервисов** — однонаправленная зависимость между IT-сервисами, указывающая, что один сервис использует другой для своей работы

- **Граф зависимостей** — визуальное представление IT-сервисов и связей между ними в виде ориентированного графа

- **Журнал действий** — компонент системы, регистрирующий действия пользователей для обеспечения аудита и контроля

- **Сессия пользователя** — период активной работы пользователя в системе от момента входа до момента выхода или истечения времени неактивности

- **Механизм аутентификации** — компонент системы, обеспечивающий проверку подлинности пользователя при входе в систему

- **Механизм авторизации** — компонент системы, определяющий права доступа пользователя к функциям и данным системы на основе его роли

- **Резервная копия** — полная копия данных системы, создаваемая для возможности восстановления в случае сбоя

## 3. Введение

### 3.1. Назначение документа

Данный документ описывает системные требования к Автоматизированной информационной системе учёта и управления IT-сервисами (ITSIMS). Документ предназначен для:

- **Системных архитекторов** — для проектирования архитектуры системы и выбора компонентов
- **Разработчиков программного обеспечения** — для понимания контекста и границ системы перед разработкой программных компонентов
- **Бизнес-аналитиков** — для связи между бизнес-требованиями заказчика и технической реализацией
- **Тестировщиков** — для составления тест-кейсов на системном уровне и проверки соответствия требованиям
- **Технического директора заказчика** — для контроля соответствия разрабатываемой системы потребностям организации

Документ служит промежуточным уровнем между Техническим заданием (бизнес-требования) и Требованиями к программному обеспечению (технические детали реализации).

### 3.2. Область применения

Документ охватывает следующие аспекты системы ITSIMS:

- Архитектуру системы и взаимодействие основных компонентов
- Поведение системы как единого целого на системном уровне
- Функциональные требования к управлению каталогом IT-сервисов, документацией и правами доступа
- Требования к безопасности системы (аутентификация, авторизация, защита данных, логирование)
- Требования к пользовательскому интерфейсу и удобству использования
- Требования к производительности, надёжности и доступности системы
- Механизмы интеграции с внешней системой учёта аппаратного обеспечения

Документ **не охватывает**:

- Детали программной реализации компонентов (см. Требования к ПО)
- Конкретные технологии, языки программирования и фреймворки (см. Требования к ПО)
- Протоколы обмена данными и форматы сообщений (см. Требования к ПО)
- Структуру базы данных, таблицы и поля (см. Требования к ПО)
- Алгоритмы реализации функций (см. Требования к ПО)

### 3.3. Связь с Техническим заданием

Данный документ детализирует требования из [Технического задания](./1_technical_specification.md) на системном уровне. Каждое требование SR связано с одним или несколькими требованиями ТЗ через таблицу трассировки [TS_to_SR](./2_TS_to_SR.md).

**Принцип трассировки:**

- Каждое требование из ТЗ раскрывается одним или несколькими системными требованиями
- Системные требования описывают **КАК** система будет реализовывать бизнес-требования на системном уровне
- Системные требования не используют конкретные технологии, но описывают механизмы и поведение системы

Трассировка обеспечивает:

- **Полноту покрытия** — все требования ТЗ детализированы в SR
- **Прозрачность связи** — можно отследить, какое требование ТЗ реализуется конкретными системными требованиями
- **Управление изменениями** — при изменении требования ТЗ можно определить, какие системные требования необходимо пересмотреть



## 4. Общее описание системы

### 4.1. Архитектура системы

Система ITSIMS реализована как веб-приложение с классической трёхуровневой архитектурой: клиентская часть, серверная часть и уровень хранения данных.

**Основные компоненты системы:**

1. **Клиентская часть (веб-интерфейс)**
   - Выполняется в веб-браузере пользователя
   - Обеспечивает визуализацию данных и взаимодействие пользователя с системой
   - Отправляет запросы к серверной части и отображает результаты

2. **Серверная часть**
   - Обрабатывает запросы от клиентской части
   - Реализует бизнес-логику системы
   - Управляет доступом к данным и обеспечивает безопасность
   - Состоит из следующих функциональных модулей:
     - **Модуль управления каталогом сервисов** — создание, редактирование, архивирование IT-сервисов, управление категориями и связями между сервисами
     - **Модуль управления документацией** — загрузка, хранение, организация и контроль доступа к файлам документации
     - **Модуль управления правами доступа** — управление пользователями, ролями и разграничение прав доступа к сервисам и документации
     - **Модуль обслуживания** — регистрация фактов обслуживания и ведение истории обслуживания сервисов
     - **Модуль визуализации связей** — построение и отображение графа зависимостей между сервисами
     - **Модуль аутентификации и безопасности** — проверка подлинности пользователей, управление сессиями, логирование действий
     - **Модуль интеграции** — взаимодействие с внешней системой учёта аппаратного обеспечения

3. **Уровень хранения данных**
   - **Хранилище данных** — обеспечивает постоянное хранение структурированных данных о сервисах, пользователях, категориях, правах доступа, записях обслуживания и настройках системы
   - **Файловое хранилище** — обеспечивает хранение файлов документации, ассоциированных с IT-сервисами

**Взаимодействие компонентов:**

Пользователь обращается к системе через веб-браузер, отправляя запросы к серверной части. Серверная часть обрабатывает запросы, проверяет права доступа, выполняет необходимые операции с данными через хранилище данных и файловое хранилище, и возвращает результаты клиентской части для отображения пользователю.

Все взаимодействие между компонентами происходит через защищённые каналы связи. Серверная часть является единственной точкой доступа к данным системы, что обеспечивает централизованный контроль безопасности.

### 4.2. Пользовательские роли

Система поддерживает четыре роли пользователей с различными уровнями доступа к функциональности и данным:

**Администратор системы:**

- **Количество:** Может быть несколько администраторов системы в организации
- **Автоматическое назначение:** Пользователь с должностью "Технический директор" автоматически получает права администратора системы
- **Права доступа:**
  - Полный контроль над всеми IT-сервисами в системе (создание, редактирование, архивирование)
  - Управление справочником категорий сервисов
  - Создание и настройка шаблонов категорий
  - Управление связями между сервисами
  - Создание новых пользователей системы
  - Назначение администраторов сервисов
  - Управление правами доступа ко всем сервисам
  - Просмотр журнала действий всех пользователей системы
  - Управление настройками системы
- **Типичные задачи:** Поддержка системы в актуальном состоянии, создание новых сервисов, распределение администраторов по сервисам, решение глобальных вопросов в масштабе множества сервисов

**Администратор сервиса:**

- **Назначение:** Назначается администратором системы для управления конкретным IT-сервисом
- **Множественность:** Один сервис может иметь нескольких администраторов; один пользователь может быть администратором нескольких сервисов
- **Права доступа:**
  - Редактирование характеристик назначенного ему IT-сервиса
  - Загрузка, удаление и организация документации сервиса
  - Пометка документов как "для пользователей" или "для обслуживания"
  - Назначение специалистов по обслуживанию и обычных сотрудников для данного сервиса
  - Управление правами доступа к документации данного сервиса
  - Просмотр журнала действий пользователей в рамках данного сервиса
  - Просмотр связей сервиса и графа зависимостей
- **Типичные задачи:** Управление документацией сервиса, распределение прав доступа к сервису между сотрудниками, поддержание актуальности информации о сервисе

**Специалист по обслуживанию:**

- **Назначение:** Назначается администратором системы или администратором сервиса для обслуживания конкретных IT-сервисов
- **Видимость:** Видит только те IT-сервисы, к которым ему предоставлены права доступа
- **Права доступа:**
  - Просмотр детальной информации о назначенных ему IT-сервисах
  - Доступ ко всей документации сервиса (включая документы для обслуживания)
  - Регистрация фактов обслуживания IT-сервисов
  - Просмотр истории обслуживания сервиса
  - Просмотр связей сервиса с другими сервисами (один уровень глубины)
  - Просмотр графа зависимостей для назначенных сервисов
- **Типичные задачи:** Обслуживание IT-сервисов согласно регламентам, регистрация выполненных работ, изучение технической документации

**Обычный сотрудник:**

- **Назначение:** Назначается администратором системы или администратором сервиса для получения доступа к документации конкретных IT-сервисов
- **Видимость:** Видит только те IT-сервисы, к которым ему предоставлены права доступа
- **Права доступа:**
  - Просмотр списка доступных ему IT-сервисов
  - Поиск и фильтрация сервисов по категории и статусу
  - Просмотр базовой информации о сервисе (название, описание, категория, статус, версия)
  - Доступ только к документам, помеченным как "для пользователей" (руководства, инструкции по установке и настройке)
  - Скачивание документации
- **Типичные задачи:** Самостоятельная установка и настройка IT-сервисов на основе документации, изучение инструкций пользователя

### 4.3. Основные сценарии использования

**Сценарий 1: Добавление нового IT-сервиса в систему**

1. Администратор системы входит в систему
2. Система отображает главную страницу с реестром IT-сервисов
3. Администратор выбирает функцию создания нового сервиса
4. Система отображает форму создания сервиса
5. Администратор заполняет обязательные характеристики: название, краткое описание, выбирает категорию из справочника, устанавливает статус
6. Администратор опционально заполняет версию ПО и идентификаторы аппаратов
7. Администратор сохраняет сервис
8. Система проверяет уникальность названия и корректность введённых данных
9. Система создаёт запись о сервисе в хранилище данных
10. Система автоматически создаёт файловый каталог для документации на основе шаблона выбранной категории (если шаблон настроен)
11. Система отображает сообщение об успешном создании сервиса
12. Администратор назначает администратора сервиса для дальнейшего управления

**Сценарий 2: Загрузка документации сервиса**

1. Администратор сервиса входит в систему
2. Система отображает список сервисов, которыми он управляет
3. Администратор выбирает нужный сервис и переходит к управлению документацией
4. Система отображает файловую структуру каталога сервиса
5. Администратор выбирает папку или создаёт новую
6. Администратор загружает файл документации
7. Система проверяет формат файла (соответствие списку поддерживаемых) и размер (не более 20 МБ)
8. Система выполняет базовое сканирование файла
9. Система сохраняет файл в файловом хранилище
10. Администратор помечает документ как "для пользователей" или "для обслуживания"
11. Система фиксирует тип документа и обновляет структуру каталога
12. Система регистрирует действие в журнале

**Сценарий 3: Поиск и получение документации обычным сотрудником**

1. Обычный сотрудник входит в систему
2. Система отображает реестр IT-сервисов, доступных данному пользователю
3. Сотрудник использует поиск по названию или фильтрацию по категории для нахождения нужного сервиса
4. Система отображает список сервисов, соответствующих критериям поиска
5. Сотрудник выбирает нужный сервис
6. Система отображает информацию о сервисе и файловый каталог документации
7. Система показывает только документы и папки, помеченные как "для пользователей"
8. Сотрудник находит нужную инструкцию и скачивает её
9. Система предоставляет файл для скачивания
10. Сотрудник использует документацию для установки и настройки сервиса

**Сценарий 4: Регистрация факта обслуживания**

1. Специалист по обслуживанию входит в систему
2. Система отображает список сервисов, доступных для обслуживания
3. Специалист выбирает сервис, по которому проведены работы
4. Специалист просматривает техническую документацию и выполняет обслуживание
5. Специалист выбирает функцию регистрации обслуживания
6. Система отображает форму записи обслуживания
7. Специалист указывает дату проведения работ, описание выполненных работ, список исполнителей
8. Система сохраняет запись обслуживания
9. Система обновляет дату последнего обслуживания для данного сервиса
10. Система регистрирует действие в журнале
11. Система отображает сообщение об успешной регистрации

**Сценарий 5: Управление правами доступа к сервису**

1. Администратор сервиса входит в систему
2. Администратор выбирает сервис для управления правами доступа
3. Система отображает список пользователей и их текущие права доступа к данному сервису
4. Администратор добавляет нового пользователя, выбирая его из списка сотрудников организации
5. Администратор назначает роль для пользователя: Специалист по обслуживанию или Обычный сотрудник
6. Система предоставляет пользователю права согласно выбранной роли
7. Система регистрирует изменение прав доступа в журнале
8. Система отображает сообщение об успешном обновлении прав

### 4.4. Ограничения и зависимости

**Внешние зависимости:**

- **Система учёта аппаратного обеспечения организации** (опциональная зависимость)
  - Система ITSIMS использует идентификаторы аппаратов из внешней системы для привязки IT-сервисов к физическому или виртуальному оборудованию
  - В первой версии системы интеграция реализована через ручной ввод идентификаторов администратором
  - Система ITSIMS сохраняет полную работоспособность при недоступности системы учёта аппаратного обеспечения, поскольку привязка осуществляется через ссылочные идентификаторы без синхронизации данных в реальном времени

**Технические ограничения:**

- Система доступна пользователям только через веб-браузер на настольных компьютерах или ноутбуках
- Мобильные устройства (смартфоны, планшеты) не поддерживаются в первой версии системы
- Максимальный размер загружаемого файла документации ограничен 20 МБ
- Максимальная длина имени файла ограничена 50 символами
- Система поддерживает ограниченный набор форматов файлов документации, настраиваемый администратором для каждого сервиса

**Организационные ограничения:**

- Все пользователи системы должны быть сотрудниками организации АО «БАРС Груп»
- Самостоятельная регистрация новых пользователей не предусмотрена — все пользователи создаются администратором системы
- Запросы на доступ к сервисам от пользователей не предусмотрены — права доступа предоставляются только администраторами

**Ограничения производительности:**

- Система рассчитана на одновременную работу не менее 3000 пользователей
- Система должна обеспечивать управление каталогом из не менее 2000 IT-сервисов
- Время загрузки главной страницы не должно превышать 2 секунд при стандартной скорости интернет-соединения 10 Мбит/с
