## 1. Цель документа

Данный документ определяет специфичные правила оформления **Технического задания (ТЗ)** в проекте ITSIMS. Документ является дополнением к [Общим правилам оформления артефактов](./general_artifact_rules.md) и содержит требования, применимые исключительно к артефакту типа "Техническое задание".



## 2. Описание документа

**Техническое задание (ТЗ)** — это основополагающий документ, который устанавливает договорённости между заказчиком и исполнителем относительно того, что должна представлять собой разрабатываемая система. ТЗ фиксирует требования заказчика на бизнес-уровне и служит отправной точкой для всех последующих этапов разработки.

**Ключевые характеристики ТЗ как документа:**

1. **Контрактная функция** — ТЗ определяет обязательства исполнителя перед заказчиком, фиксирует согласованный объём работ и критерии приёмки результата

2. **Бизнес-ориентированность** — ТЗ описывает систему на языке, понятном заказчику, избегая технического жаргона и деталей реализации

3. **Высокий уровень абстракции** — ТЗ отвечает на вопрос "**ЧТО** должна делать система", не вдаваясь в вопросы "**КАК** это будет реализовано"

4. **Основа для детализации** — ТЗ служит источником для создания более детальных документов (требования к системе, требования к ПО), каждое требование ТЗ впоследствии раскрывается и конкретизируется

5. **Стабильность** — ТЗ изменяется значительно реже, чем технические документы, так как бизнес-требования заказчика более устойчивы, чем технические решения

**Принцип:** Техническое задание отвечает на вопрос "**ЧТО** должна делать система с точки зрения бизнеса", не вдаваясь в технические детали реализации.



## 3. Структура ТЗ

ТЗ **должно** содержать следующие разделы в указанном порядке:

```markdown
## 1. Цель документа
## 2. Термины, определения и сокращения
## 3. Общие сведения
## 4. Назначение и цели создания системы
## 5. Требования к системе
   ### 5.1. Функциональные требования
   ### 5.2. Требования к интеграциям
   ### 5.3. Нефункциональные требования
## 6. Связанные документы
```

### 3.1 Пояснения к разделам

#### Раздел 1: Цель документа

См. [раздел 2.5](./general_artifact_rules.md#25-раздел-цель-документа) общих правил оформления артефактов.

#### Раздел 2: Термины, определения и сокращения

Подробное описание структуры и правил оформления этого раздела см. в [разделе 2.6](./general_artifact_rules.md#26-раздел-термины-определения-и-сокращения) общих правил оформления артефактов.

#### Раздел 3: Общие сведения

Содержит контекстную информацию о проекте:
- Полное наименование системы
- Сведения о заказчике
- Сроки выполнения работ
- Основания для разработки

#### Раздел 4: Назначение и цели создания системы

Описывает:
- Основные цели создания системы
- Бизнес-задачи, которые решает система
- Ожидаемые результаты от внедрения

#### Раздел 5: Требования к системе

Основной раздел ТЗ, содержащий требования трёх типов:

**5.1. Функциональные требования** (`TS-F001`, `TS-F002`, ...)
- Описывают функции, которые должна выполнять система
- Формулируются с точки зрения пользователя
- Не содержат технических решений

**5.2. Требования к интеграциям** (`TS-I001`, `TS-I002`, ...)
- Описывают необходимые интеграции с внешними системами
- На уровне "с какими системами интегрироваться", без протоколов и API

**5.3. Нефункциональные требования** (`TS-NF001`, `TS-NF002`, ...)
- Производительность
- Надёжность
- Масштабируемость
- Безопасность (на бизнес-уровне)



## 4. Работа с требованиями в ТЗ

### 4.1 Общий формат

Оформление требований в ТЗ основано на правилах из раздела 4 общего документа [general_artifact_rules.md](./general_artifact_rules.md#41-форматирование-требований): каждое требование **должно** быть оформлено следующим образом:

```markdown
- <a id="<req-id>"></a>**TS-<тип><номер>:** <Формулировка требования> [↓](...)
```

**Компоненты структуры требования:**

| Компонент               | Описание                                                                                                                       | Примеры                     |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------ | --------------------------- |
| `<a id="req-id"></a>`   | HTML-якорь для навигации из таблиц трассировки                                                                                 | `<a id="ts-f001"></a>`      |
| **Тип**                 | Тип требования (функциональное, нефункциональное, интеграционное)                                                              | `F`, `NF`, `I`              |
| **Номер**               | Трёхзначный порядковый номер                                                                                                   | `001`, `042`, `123`         |
| Формулировка требования | Текст требования (см. [раздел 4.2 общих правил оформления артефактов](./general_artifact_rules.md#42-формулировка-требований)) | Система должна ...          |
| `[↓](...)`              | Ссылка на таблицу трассировки требования вниз                                                                                  | `[↓](./2_TS_to_SR#ts-f001)` |

**Типы требований и префиксы**

| Тип требования              | Префикс | Пример ID  |
| --------------------------- | ------- | ---------- |
| Функциональное требование   | `F`     | `TS-F001`  |
| Нефункциональное требование | `NF`    | `TS-NF001` |
| Требование к интеграции     | `I`     | `TS-I001`  |

### 4.2 Правила формулировки требований

**Обязательные правила:**

1. **Язык формулировки:** "Система должна..."
2. **Бизнес-ориентированность:** Требование описывает бизнес-функцию, а не техническую реализацию
3. **Запрет технических терминов:** Не используются названия технологий, протоколов, форматов данных
4. **Понятность для заказчика:** Заказчик должен понимать требование без технических знаний
5. **Атомарность:** Одно требование = одна функция/характеристика

**Примеры корректных требований ТЗ:**

```markdown
- **TS-F001:** Система должна обеспечивать возможность добавления нового IT-сервиса в реестр
- **TS-F002:** Система должна обеспечивать возможность редактирования информации об IT-сервисе
- **TS-F003:** Система должна обеспечивать поиск IT-сервиса по названию и категории
- **TS-NF001:** Система должна поддерживать одновременную работу не менее 50 пользователей
- **TS-I001:** Система должна интегрироваться с существующей системой единого входа организации
```

**Примеры некорректных требований (слишком технические для ТЗ):**

```markdown
❌ TS-F001: Система должна предоставлять REST API endpoint для создания сервиса
❌ TS-F002: Данные о сервисах должны храниться в реляционной БД PostgreSQL
❌ TS-NF001: Время ответа API не должно превышать 200мс при нагрузке 100 RPS
```

*Эти требования правильны для "Требований к системе" или "Требований к ПО", но не для ТЗ.*

## 5. Что ЗАПРЕЩЕНО в Техническом задании

**В ТЗ категорически запрещено:**

- ❌ Упоминание технологий, фреймворков, языков программирования
- ❌ Описание структуры базы данных, таблиц, полей
- ❌ Спецификация API (endpoints, методы, коды ответов)
- ❌ Описание архитектурных решений (микросервисы, монолит, брокеры сообщений)
- ❌ Указание протоколов обмена данными (HTTP, WebSocket, gRPC)
- ❌ Форматы данных (JSON, XML, Protobuf)
- ❌ Детали реализации интерфейса (компоненты, React, Vue)

**Правильный уровень детализации:**

- ✅ "Система должна обеспечивать возможность поиска сервисов"
- ❌ "Система должна предоставлять REST API для поиска через GET /api/services?query=..."

**Технические детали принадлежат документам:**
- Требования к системе (SR) — для системного уровня
- Требования к ПО (SWR) — для технической реализации

Данная таблица обобщает ключевые параметры Технического задания для быстрого ориентирования:

| Параметр                  | Значение                                           |
| ------------------------- | -------------------------------------------------- |
| **Префикс документа**     | `TS` (Technical Specification)                     |
| **Уровень детализации**   | Высокоуровневый (бизнес)                           |
| **Целевая аудитория**     | Заказчик, бизнес-аналитики, менеджеры проекта      |

| **Трассировка вверх**     | —                                                  |
| **Трассировка вниз**      | К Требованиям к системе (`SR-`)                    |
