## 1. Цель документа

Данный документ определяет специфичные правила оформления **Требований к программному обеспечению (Software Requirements, SWR)** в проекте ITSIMS. Документ является дополнением к [Общим правилам оформления артефактов](./general_artifact_rules.md) и содержит требования, применимые исключительно к артефакту типа "Требования к программному обеспечению".



## 2. Описание документа

**Требования к программному обеспечению (Software Requirements, SWR)** — это документ технического уровня, который описывает детальную реализацию программных компонентов системы. SWR отвечает на вопрос "**КАК** будет реализовано программное обеспечение для выполнения системных требований".

**Ключевые характеристики SWR как документа:**

1. **Техническая детализация SR** — каждое требование SWR раскрывает одно или несколько требований из Системных требований, детализируя их на техническом уровне

2. **Компонентная ориентированность** — SWR описывает систему через призму технических компонентов (Backend, Frontend, Database, Infrastructure), а не функциональных блоков

3. **Максимальный уровень детализации** — SWR содержит конкретные технологии, протоколы, форматы данных, структуры БД, спецификации API

4. **Основа для разработки** — SWR служит непосредственным руководством для программистов, содержит всё необходимое для реализации

5. **Конечный уровень трассировки** — SWR связан только с вышестоящим документом (SR), трассировка вниз отсутствует

6. **Расширенная типизация требований** — SWR использует специфичные категории требований (API, DB, UI, DEP, CONFIG, LOG, TEST)

**Принцип:** Требования к программному обеспечению отвечают на вопрос "**КАК** технически будет реализована система", описывая конкретные технологии, протоколы, структуры данных и архитектурные решения.



## 3. Структура Software Requirements

### 3.1 Обязательные разделы

SWR **должен** содержать следующие разделы в указанном порядке:

```markdown
## 1. Цель документа
## 2. Термины, определения и сокращения
## 3. Архитектура программного обеспечения
   ### 3.1. Компоненты системы
   ### 3.2. Взаимодействие компонентов
   ### 3.3. Технологический стек
## 4. Требования к Backend
   ### 4.1. API модуля "[Модуль 1]"
   ### 4.2. API модуля "[Модуль 2]"
   ### 4.N. API модуля "[Модуль N]"
## 5. Требования к Frontend
   ### 5.1. Структура компонентов
   ### 5.2. Управление состоянием
   ### 5.3. Требования к UI компонентам
## 6. Требования к данным
   ### 6.1. Схема базы данных
   ### 6.2. Миграции и хранение
## 7. Требования к инфраструктуре
   ### 7.1. Развёртывание
   ### 7.2. Конфигурация
   ### 7.3. Логирование и мониторинг
## 8. Требования к тестированию
## 9. Связанные документы
```

**Важное отличие от ТЗ и SR:** Структура SWR ориентирована на **технические компоненты** системы, а не на функциональные блоки. Это обеспечивает удобство использования документа разработчиками.

### 3.2 Пояснения к разделам

#### Раздел 1: Цель документа

См. [раздел 2.5](./general_artifact_rules.md#25-раздел-цель-документа) общих правил оформления артефактов.

#### Раздел 2: Термины, определения и сокращения

Подробное описание структуры и правил оформления этого раздела см. в [разделе 2.6](./general_artifact_rules.md#26-раздел-термины-определения-и-сокращения) общих правил оформления артефактов.

**Важное замечание для SWR:** В этом разделе **переиспользуются** базовые термины из ТЗ и SR, и добавляются технические термины, связанные с используемыми технологиями, протоколами и инструментами.

#### Раздел 3: Архитектура программного обеспечения

Описывает общую структуру системы на техническом уровне:

**3.1. Компоненты системы:**
- Перечень основных программных модулей и их назначение
- Используемые технологии для каждого компонента
- Зависимости между компонентами

**3.2. Взаимодействие компонентов:**
- Протоколы обмена данными между компонентами (REST, gRPC, WebSocket)
- Форматы передаваемых данных (JSON, Protobuf)
- Диаграммы взаимодействия (опционально)

**3.3. Технологический стек:**
- Языки программирования и версии
- Фреймворки и библиотеки
- СУБД и системы хранения
- Инструменты разработки и CI/CD

#### Раздел 4: Требования к Backend

Основной раздел документа, структурированный **по модулям API**.

**Структура описания модуля:**

```markdown
### 4.1. API модуля "[Название]"

#### 4.1.1. Назначение
Краткое описание функций модуля

#### 4.1.2. Endpoints
Детальное описание REST API endpoints с примерами

#### 4.1.3. Валидация и обработка ошибок
Правила валидации входных данных, коды ответов

#### 4.1.4. Безопасность
Требования к аутентификации и авторизации для модуля
```

#### Раздел 5: Требования к Frontend

**5.1. Структура компонентов:**
- Иерархия React/Vue/Angular компонентов
- Переиспользуемые компоненты
- Макеты страниц

**5.2. Управление состоянием:**
- Технология (Redux, MobX, Zustand, Pinia)
- Структура store
- Асинхронные операции

**5.3. Требования к UI компонентам:**
- Требования к конкретным компонентам интерфейса
- Адаптивность и breakpoints
- Доступность (a11y)

#### Раздел 6: Требования к данным

**6.1. Схема базы данных:**
- Описание таблиц с полями, типами, ограничениями
- Связи между таблицами (foreign keys)
- Индексы для оптимизации

**6.2. Миграции и хранение:**
- Стратегия миграций
- Резервное копирование
- Требования к производительности БД

#### Раздел 7: Требования к инфраструктуре

**7.1. Развёртывание:**
- Контейнеризация (Docker)
- Оркестрация (Kubernetes, Docker Compose)
- Окружения (dev, staging, production)

**7.2. Конфигурация:**
- Переменные окружения
- Секреты и их хранение
- Конфигурационные файлы

**7.3. Логирование и мониторинг:**
- Формат логов
- Уровни логирования
- Метрики и алерты

#### Раздел 8: Требования к тестированию

- Требования к unit-тестам
- Требования к интеграционным тестам
- Требования к e2e тестам
- Покрытие кода



## 4. Работа с требованиями в SWR

### 4.1 Общий формат

Оформление требований в SWR основано на правилах из раздела 4 общего документа [general_artifact_rules.md](./general_artifact_rules.md#41-форматирование-требований): каждое требование **должно** быть оформлено следующим образом:

```markdown
- <a id="<req-id>"></a>**SWR-<тип><номер>:** <Детальная техническая формулировка требования> [↑](...)
```

**Компоненты структуры требования:**

| Компонент               | Описание                                                                 | Примеры                                |
| ----------------------- | ------------------------------------------------------------------------ | -------------------------------------- |
| `<a id="req-id"></a>`   | HTML-якорь для навигации из таблиц трассировки                           | `<a id="swr-api001"></a>`              |
| **Тип**                 | Тип требования (API, DB, UI и т.д.)                                      | `API`, `DB`, `UI`, `DEP`               |
| **Номер**               | Трёхзначный порядковый номер                                             | `001`, `042`, `123`                    |
| Формулировка требования | Текст требования (см. [раздел 4.3](#43-правила-формулировки-требований)) | Модуль должна ...                      |
| `[↑](...)`              | Ссылка на таблицу трассировки требования вверх (к SR)                    | `[↑](../4_SR_to_SWR.md#swr-api001-up)` |

**Примеры требований с полной структурой:**

```markdown
- <a id="swr-api001"></a>**SWR-API001:** Модуль "Управление сервисами" должен предоставлять REST API endpoint POST /api/v1/services для создания нового сервиса [↑](../4_SR_to_SWR.md#swr-api001-up)

- <a id="swr-db001"></a>**SWR-DB001:** Данные о сервисах должны храниться в таблице services с полями: id (serial primary key), name (varchar 255 not null unique), status (varchar 20 not null) [↑](../4_SR_to_SWR.md#swr-db001-up)
```

### 4.2 Типы требований и префиксы

| Тип требования                     | Префикс  | Пример ID       | Описание                                    |
| ---------------------------------- | -------- | --------------- | ------------------------------------------- |
| Функциональное требование к модулю | `F`      | `SWR-F001`      | Общие функциональные требования к модулям   |
| Требование к API                   | `API`    | `SWR-API001`    | REST/GraphQL endpoints, форматы запросов    |
| Требование к базе данных           | `DB`     | `SWR-DB001`     | Таблицы, поля, индексы, миграции            |
| Требование к UI компоненту         | `UI`     | `SWR-UI001`     | React/Vue компоненты, стили, адаптивность   |
| Нефункциональное требование        | `NF`     | `SWR-NF001`     | Производительность, надёжность на техуровне |
| Требование безопасности            | `SEC`    | `SWR-SEC001`    | JWT, хеширование, CORS, валидация           |
| Требование к развёртыванию         | `DEP`    | `SWR-DEP001`    | Docker, Kubernetes, CI/CD                   |
| Требование к тестированию          | `TEST`   | `SWR-TEST001`   | Unit, integration, e2e тесты                |
| Требование к конфигурации          | `CONFIG` | `SWR-CONFIG001` | Переменные окружения, настройки             |
| Требование к логированию           | `LOG`    | `SWR-LOG001`    | Формат логов, уровни, хранение              |

### 4.3 Правила формулировки требований

**Обязательные правила:**

1. **Язык формулировки:** "Модуль должен...", "Компонент должен...", "API должен..."
2. **Максимальная детализация:** Требование содержит все технические детали для реализации
3. **Конкретные технологии:** Указываются используемые технологии, протоколы, форматы
4. **Привязка к компонентам:** Требование явно указывает, какой модуль/компонент должен реализовать функциональность
5. **Тестируемость:** Требование должно быть однозначно проверяемым
6. **Детализация SR:** Каждое SWR-требование детализирует требование из System Requirements

**Примеры корректных требований SWR:**

```markdown
- <a id="swr-api001"></a>**SWR-API001:** Модуль "Управление сервисами" должен предоставлять REST API endpoint POST /api/v1/services для создания нового сервиса с JSON payload: { name: string (required, max 255), description: string (optional), category: string (required), status: enum [planned, active, retired], owner_id: integer (required) } [↑](../4_SR_to_SWR.md#swr-api001-up)

- <a id="swr-api002"></a>**SWR-API002:** API должен возвращать HTTP 201 Created при успешном создании сервиса с телом ответа: { id: integer, created_at: ISO 8601 timestamp } [↑](../4_SR_to_SWR.md#swr-api002-up)

- <a id="swr-db001"></a>**SWR-DB001:** Таблица services должна содержать поля: id (serial primary key), name (varchar 255 not null unique), description (text), category (varchar 100 not null), status (varchar 20 not null check in [planned, active, retired]), owner_id (integer not null references users.id), created_at (timestamp default now()), updated_at (timestamp default now()) [↑](../4_SR_to_SWR.md#swr-db001-up)

- <a id="swr-sec001"></a>**SWR-SEC001:** Пароли пользователей должны храниться в виде хеша bcrypt с cost factor 12 [↑](../4_SR_to_SWR.md#swr-sec001-up)
```

**Сравнение уровней детализации:**

| Уровень     | Пример требования                                                                                                                                              |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ТЗ (TS)** | Система должна обеспечивать возможность добавления нового IT-сервиса в реестр                                                                                  |
| **SR**      | Система должна обеспечивать создание записи IT-сервиса с указанием атрибутов: название, категория, описание, владелец, статус                                  |
| **SWR**     | Модуль "Управление сервисами" должен предоставлять REST API endpoint POST /api/v1/services для создания нового сервиса с JSON payload: { name, category, ... } |

### 4.4 Нумерация требований

- Нумерация **независимая** для каждого типа требований
- Формат: **трёхзначное число с ведущими нулями** (`001`, `002`, ..., `010`, ...)
- Пример последовательности: `SWR-API001`, `SWR-API002`, ..., `SWR-API010`, ...

**Правила нумерации при изменениях:**

- При удалении требования его номер **не переиспользуется** (пропуск в нумерации допустим)
- Новые требования получают **следующий свободный номер** в последовательности
- Изменение формулировки **не изменяет номер** требования

### 4.5 Трассировка требований

Каждое требование SWR **обязательно** должно иметь трассировку вверх (к SR). Трассировка вниз отсутствует — SWR является самым нижним уровнем требований.

**Формат трассировки:**

```markdown
- <a id="swr-api001"></a>**SWR-API001:** Модуль должен предоставлять REST API endpoint POST /api/v1/services [↑](../4_SR_to_SWR.md#swr-api001-up)
```

**Правила трассировки:**

- Ссылка `[↑]` **обязательна** для всех требований SWR
- Ссылка ведёт на строку в таблице трассировки `4_SR_to_SWR.md`
- Если требование SR не существует — это **ошибка**, каждое SWR должно покрывать требование SR
- Допускается множественная трассировка вверх (несколько SR могут породить одно SWR)

### 4.6 Примеры оформления разделов SWR

#### Пример 1: Требования к Backend (API модуля)

```markdown
## 4. Требования к Backend

### 4.1. API модуля "Управление сервисами"

#### 4.1.1. Назначение

Модуль отвечает за CRUD операции с IT-сервисами через REST API.

#### 4.1.2. Endpoints

##### POST /api/v1/services — Создание сервиса

- <a id="swr-api001"></a>**SWR-API001:** Модуль должен предоставлять endpoint POST /api/v1/services для создания сервиса с JSON payload: { name: string (required, max 255), description: string (optional), category: string (required), status: enum [planned, active, retired], owner_id: integer (required) } [↑](../4_SR_to_SWR.md#swr-api001-up)

- <a id="swr-api002"></a>**SWR-API002:** При успешном создании API должен возвращать HTTP 201 с телом: { id: integer, created_at: ISO 8601 } [↑](../4_SR_to_SWR.md#swr-api002-up)

- <a id="swr-api003"></a>**SWR-API003:** При ошибке валидации API должен возвращать HTTP 400 с телом: { error: string, details: array } [↑](../4_SR_to_SWR.md#swr-api003-up)

#### 4.1.3. Валидация

- <a id="swr-f001"></a>**SWR-F001:** Модуль должен валидировать поле name: обязательное, максимум 255 символов, только буквенно-цифровые символы, пробелы и дефисы [↑](../4_SR_to_SWR.md#swr-f001-up)
```

## 5. Что ОБЯЗАТЕЛЬНО в SWR

В SWR **обязательно указывать:**

- ✅ Конкретные технологии и версии (React 18, Node.js 20, PostgreSQL 16)
- ✅ Спецификацию API (endpoints, методы, параметры, форматы запросов и ответов)
- ✅ Структуру базы данных (таблицы, поля, типы, индексы, ограничения)
- ✅ Форматы данных (JSON, XML, Protobuf) с примерами
- ✅ Протоколы обмена (REST, gRPC, WebSocket)
- ✅ Коды ответов HTTP и обработку ошибок
- ✅ Валидацию данных с конкретными правилами
- ✅ Названия библиотек и фреймворков с версиями
- ✅ Архитектуру компонентов UI
- ✅ Требования к развёртыванию и конфигурации
- ✅ Трассировку к требованиям SR

**Допускается (по необходимости):**

- ✅ Примеры JSON payload (для иллюстрации форматов)
- ✅ Примеры SQL-запросов (для сложных случаев)
- ✅ Диаграммы последовательности и классов
- ✅ OpenAPI/Swagger спецификации (встроенные или ссылки)
- ✅ Схемы баз данных (ERD)
- ✅ Конфигурационные примеры



## 6. Что ЗАПРЕЩЕНО в SWR

**В SWR категорически запрещено:**

- ❌ **Повторение информации из SR:** Не дублировать общее описание системы, введение, пользовательские роли — эта информация уже есть в System Requirements
- ❌ **Абстрактные формулировки:** "Система должна быть быстрой", "Данные должны храниться безопасно" — без конкретных технических деталей
- ❌ **Требования без трассировки:** Каждое требование SWR обязательно должно ссылаться на требование SR
- ❌ **Бизнес-формулировки:** "Пользователь должен иметь возможность..." — это уровень ТЗ/SR, в SWR формулируем через модули и компоненты
- ❌ **Неполные технические спецификации:** Указание технологии без версии, endpoint без формата запроса/ответа

**Правильный уровень детализации:**

```markdown
✅ SWR уровень: Модуль должен предоставлять REST API endpoint POST /api/v1/services с JSON payload
❌ Слишком абстрактно (уровень SR): Система должна обеспечивать создание IT-сервиса

✅ SWR уровень: Пароли должны храниться в виде хеша bcrypt с cost factor 12
❌ Слишком абстрактно (уровень SR): Система должна хранить пароли в защищённом виде

✅ SWR уровень: Таблица services должна иметь индекс на поле category для оптимизации поиска
❌ Слишком абстрактно: Данные должны храниться эффективно
```



## 7. Справочная информация по SWR

Данная таблица обобщает ключевые параметры Требований к программному обеспечению:

| Параметр                  | Значение                                                                 |
| ------------------------- | ------------------------------------------------------------------------ |
| **Префикс документа**     | `SWR` (Software Requirements)                                            |
| **Уровень детализации**   | Технический (максимальный)                                               |
| **Целевая аудитория**     | Разработчики, тестировщики, DevOps-инженеры                              |
| **Ключевой вопрос**       | КАК технически будет реализована система?                                |
| **Технические детали**    | Обязательны (технологии, протоколы, форматы, версии)                     |
| **Структура документа**   | По техническим компонентам (Backend, Frontend, Database, Infrastructure) |
| **Префикс ID требований** | `SWR-` (например: `SWR-API001`, `SWR-DB001`, `SWR-UI001`, `SWR-DEP001`)  |
| **Категории требований**  | F, API, DB, UI, NF, SEC, DEP, TEST, CONFIG, LOG                          |
| **Трассировка вверх**     | К Требованиям к системе (`SR-`)                                          |
| **Трассировка вниз**      | Отсутствует (конечный уровень)                                           |
