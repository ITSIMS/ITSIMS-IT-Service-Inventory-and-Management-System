## 1. Введение

### 1.1. Цель документа

Настоящий документ определяет процессы, правила и ответственность за управление конфигурацией проекта ITSIMS.

План КУ обеспечивает:
- Контроль над идентификацией, изменением и хранением артефактов проекта
- Отслеживаемость изменений
- Возможность восстановления конкретных версий документов

### 1.2. Область применения

Документ распространяется на все артефакты проекта:
- Техническое задание и требования к системе
- Планы и правила работы с проектом
- Отчёты валидации и инспекций
- Задачи и отчёты о работе команды

На текущем этапе проект является документационным. При переходе к разработке ПО план будет дополнен разделами по управлению исходным кодом.

## 7. Базовые версии

Базовые версии артефактов фиксируются через таблицу версионирования в начале каждого документа.

**Формат записи:**
- Дата изменения
- Номер версии
- Автор
- Ссылка на задачу
- Комментарий

**Правила версионирования (формат vM.m):**
- Мажорная версия (v2.0) — значительная переработка структуры или содержания
- Минорная версия (v2.1) — добавление новых разделов, дополнения, исправления

### 7.1. Базовые линии этапа верификации

Для этапа верификации фиксируются следующие baseline:

| ID baseline | Дата фиксации | Состав baseline | Ответственный за фиксацию |
| ----------- | ------------- | --------------- | ------------------------- |
| BL-VER-2026-02-A | 2026-02-19 | `candidate_1_dependency_graph.md`, `candidate_2_rbac.md` | Носов А.А. |
| BL-VER-2026-02-B | 2026-02-20 | `planning/repository/verification_git_regulation.md`, `validations/6_verification_candidates/2026-02-19_verification_candidates.md` | Носов А.А. |

Baseline фиксируется после выполнения ревью и до начала демонстрации преподавателю.

### 7.2. Версионирование тестовых материалов

Для тестовых материалов (сценарии, чеклисты, отчеты демонстрации) действует правило:

- формат имени файла: `<YYYY-MM-DD>_<artifact>.md`;
- первая поставка: `v1.0`;
- изменение шагов сценариев без смены охвата: `v1.1`, `v1.2` и т.д.;
- добавление новых групп сценариев (новый блок проверок): следующая мажорная версия;
- в каждой версии обязательно указывать ссылку на задачу в таблице изменений документа.



## 8. Управление изменениями

Упрощённый процесс внесения изменений в КЕ:

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  Инициация  │ →  │ Реализация  │ → │  Проверка   │ → │  Принятие   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

### Этапы

1. **Инициация**
   - Создание задачи в `task-tracker/`
   - Создание ветки в Git (согласно [git_workflow.md](../repository/git_workflow.md))

2. **Реализация**
   - Работа в отдельной ветке
   - Регулярные коммиты с описательными сообщениями

3. **Проверка**
   - Ревью тимлидом (Баранов А.Т.)
   - Или ответственным за верификацию (Зацепилин А.В.)

4. **Принятие**
   - Merge в ветку main
   - Обновление таблицы версионирования в документе
   - Перемещение задачи в `3_done/`

### 8.1. Согласование изменений по классам КЕ

| Класс КЕ | Кто инициирует изменение | Кто согласует | Критерий принятия |
| -------- | ------------------------ | ------------- | ----------------- |
| Артефакты кандидатов | Вальковец Д.И. | Зацепилин А.В., Баранов А.Т. | Трассировка требований и прохождение review |
| Тестовые материалы верификации | Вальковец Д.И., Зацепилин А.В. | Баранов А.Т. | Воспроизводимость сценариев и проверяемые ожидания |
| Процессы Git/PR | Гречко И.В. | Баранов А.Т. | Соответствие workflow и регламенту верификации |
| План КУ и baseline | Носов А.А. | Баранов А.Т. | Полнота реестра КЕ и актуальность версий |

### 9.2. Выходы процесса верификации

Выход процесса верификации — отчёт по кандидату, заполненный по шаблону [\_verification_candidates_report_template.md](../../validations/6_verification_candidates/_verification_candidates_report_template.md). Выходы должны допускать однозначную интерпретацию.

#### 9.2.1. Итоговое заключение

Допустимы **только** три значения (выбирается ровно одно):

| Значение | Интерпретация |
| -------- | ------------- |
| **Принять без замечаний** | Все критерии проверки выполнены полностью (`+`), дефектов нет, рисков нет. Кандидат готов к демонстрации. |
| **Принять с доработками** | Есть дефекты не выше приоритета Medium и/или критерии выполнены частично (`+/-`); утверждён план доработок с назначенными сроками и ответственными. Кандидат может быть продемонстрирован после выполнения плана доработок. |
| **Отклонить** | Есть критичные дефекты (приоритет High) без митигирующих действий, или критерии не выполнены (`-`). Кандидат не готов к демонстрации; требуется переработка. |

#### 9.2.2. Статусы дефектов

Допустимы **только** следующие значения:

| Статус | Определение |
| ------ | ----------- |
| **Открыт** | Дефект зафиксирован, исправление не начато |
| **В работе** | Исправление выполняется |
| **Закрыт** | Дефект исправлен и проверен |
| **Отложен** | Исправление отложено по решению команды с указанием причины |

#### 9.2.3. Критерий полноты выхода

Отчёт считается полным и выход процесса — сформированным, если:

- заполнены все обязательные поля шаблона;
- итоговое заключение выбрано из трёх допустимых значений;
- для каждого дефекта указан статус из закрытого списка;
- при выборе «Принять с доработками» заполнена таблица плана доработок с полями: №, Доработка, Срок, Ответственный, Статус.
