## 1. Цель документа

Данный документ определяет обязательные и рекомендуемые элементы структуры документов проекта ITSIMS. Цель — обеспечить единообразие и читаемость всех документов в репозитории.

**Принцип:** Структура документа должна быть предсказуемой и помогать читателю быстро находить нужную информацию.



## 2. Обязательные элементы документа

Каждый документ проекта **обязательно** содержит следующие элементы в указанном порядке:

### 2.1 Заголовок документа

**Расположение:** Самое начало документа (первая строка).

**Формат:** Заголовок уровня 1 (`# Название документа`).

**Правила:**
- Заголовок точно описывает содержание документа
- Пишется без точки в конце
- Использует русский язык (если документ на русском)
- Краткий и информативный (2-7 слов)

**Примеры:**
- `# Техническое задание`
- `# План конфигурационного управления`
- `# Отчёт о формальной инспекции спецификации`
- `# Общие принципы оформления документов`

### 2.2 Таблица версионирования

**Расположение:** Первый элемент документа (сразу после заголовка документа).

**Формат:**

```markdown
| Дата       | Номер версии | Автор   | Ссылка на задачу                                      | Комментарий           |
| ---------- | ------------ | ------- | ----------------------------------------------------- | --------------------- |
| ГГГГ-ММ-ДД | vM.m         | Фамилия | [Название задачи](task-tracker/3_done/ГГГГ-ММ-ДД_...) | Кратко что поменялось |
```

**Правила заполнения:**

- **Дата** — дата внесения изменений в формате ГГГГ-ММ-ДД
- **Номер версии** — семантическое версионирование (vМ.m):
  - Увеличивай мажорную версию (M) при значительных изменениях структуры или содержания
  - Увеличивай минорную версию (m) при небольших правках, дополнениях, исправлениях
- **Автор** — фамилия автора изменений
- **Ссылка на задачу** — относительная ссылка на файл задачи в `task-tracker/`
- **Комментарий** — краткое описание внесённых изменений (1-3 слова)

**Пример:**

```markdown
| Дата       | Номер версии | Автор     | Ссылка на задачу                                                     | Комментарий             |
| ---------- | ------------ | --------- | -------------------------------------------------------------------- | ----------------------- |
| 2025-11-28 | v1.0         | Вальковец | [2025-11-27_Valkovets.md](../../task-tracker/3_done/2025-11-27_...)  | Первая версия документа |
| 2025-12-03 | v1.1         | Зацепилин | [2025-12-02_Zatsepilin.md](../../task-tracker/3_done/2025-12-02_...) | Исправление опечаток    |
| 2025-12-05 | v2.0         | Вальковец | [2025-12-04_Valkovets.md](../../task-tracker/3_done/2025-12-04_...)  | Добавлен раздел API     |
```



### 2.3 Раздел с описанием цели документа

**Расположение:** Сразу после таблицы версионирования.

**Формат:** Раздел уровня 2 (`## 1. Цель документа` или `## Описание`).

**Содержание:**
- Краткое описание назначения документа (1-3 абзаца)
- Кому предназначен документ
- Что читатель узнает из документа

**Пример:**

```markdown
## 1. Цель документа

Данный документ определяет требования к системе управления инвентарём IT-сервисов (ITSIMS). 
Документ предназначен для разработчиков системы, тестировщиков и заказчика.

Читатель узнает функциональные и нефункциональные требования к системе, её архитектуру 
и интеграционные возможности.
```

### 2.4 Основное содержание

**Структура:** Зависит от типа документа, но должна быть логичной и последовательной.

**Общие правила:**
- Используй заголовки уровня 2 для основных разделов
- Используй заголовки уровня 3 для подразделов
- Не пропускай уровни иерархии
- Нумеруй разделы для удобства навигации (`## 2. Название раздела`)

### 2.5 Раздел "Связанные документы"

**Расположение:** Последний раздел документа.

**Формат:** Раздел уровня 2 (`## N. Связанные документы`).

**Содержание:**
- Список ссылок на связанные документы проекта
- Используй маркированный список
- Каждая ссылка сопровождается кратким описанием

**Пример:**

```markdown
## 10. Связанные документы

- [Техническое задание](../../artifacts/1_technical_specification.md) — основной документ с требованиями к системе
- [План конфигурационного управления](../planning/configuration_management_plan.md) — процессы управления изменениями
- [Выбор формата документа](./format_selection.md) — правила выбора формата документов
```

**Примечание:** Если связанных документов нет, раздел можно опустить.



## 3. Рекомендуемые элементы

Следующие элементы рекомендуется включать в документы при необходимости:

### 3.1 Оглавление

**Когда использовать:**
- Документ содержит более 5 основных разделов (уровень 2)
- Документ длиннее 200 строк
- Документ имеет сложную структуру с множеством подразделов

**Расположение:** После раздела с описанием цели, перед основным содержанием.

**Формат:** Можно использовать автоматическую генерацию оглавления в Markdown.

### 3.2 Глоссарий

**Когда использовать:**
- Документ содержит множество специфичных терминов
- Документ предназначен для аудитории с разным уровнем экспертизы

**Расположение:** В конце документа, перед разделом "Связанные документы".

### 3.3 Приложения

**Когда использовать:**
- Есть дополнительные материалы, которые не вписываются в основное содержание
- Нужно вынести большие таблицы, схемы, примеры кода

**Расположение:** После основного содержания, перед "Связанными документами".
