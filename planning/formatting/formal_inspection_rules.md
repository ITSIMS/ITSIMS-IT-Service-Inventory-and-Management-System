## 1. Цель документа

Данный документ определяет правила оформления протоколов формальной инспекции документов проекта ITSIMS. Цель — обеспечить единообразие и полноту отчётов о проверке качества артефактов.

**Принцип:** Протокол инспекции должен быть структурированным, объективным и пригодным для отслеживания исправлений.



## 2. Формат протоколов

Все протоколы формальной инспекции создаются в формате **Markdown (`.md`)**.

**Обоснование выбора Markdown:**
- ✅ Удобство работы с таблицами критериев проверки
- ✅ Простота редактирования и внесения правок
- ✅ Понятные изменения в Git diff
- ✅ Легко читается на GitHub без компиляции
- ✅ При необходимости конвертируется в PDF через Pandoc



## 3. Структура протокола формальной инспекции

Каждый протокол **обязательно** содержит следующие разделы в указанном порядке:

### 3.1 Заголовок документа

```markdown
# Отчёт о формальной инспекции <Название артефакта>
```

**Примеры:**
- `# Отчёт о формальной инспекции ТЗ`
- `# Отчёт о формальной инспекции таблицы трассировки`

### 3.2 Метаданные инспекции

```markdown
**Версия артефакта:** <номер версии>

**Дата инспекции:** <ДД.ММ.ГГГГ>
```

**Правила:**
- Указывается версия проверяемого артефакта (из версионной таблицы)
- Дата инспекции — день проведения проверки

### 3.3 Участники процесса

Таблица с указанием ролей и ФИО участников:

```markdown
| Роль                | ФИО |
| ------------------- | --- |
| **Автор документа** |     |
| **Инспектор**       |     |
| **Эксперт**         |     |
| **Модератор**       |     |
```

**Правила:**
- Обязательные роли: Автор документа, Инспектор
- Опциональные роли: Эксперт, Модератор (если применимо)
- ФИО указываются в формате «Фамилия И.О.»

### 3.4 Идентификация объекта инспекции

```markdown
**Объект инспекции (ссылка на репозиторий):** <URL репозитория>

**Версия объекта (хэш коммита / тег):** <git hash>

**Состояние объекта в жизненном цикле (готов к инспекции):** <статус>
```

**Правила:**
- Ссылка на репозиторий — полный URL на GitHub
- Хэш коммита — короткий хэш (первые 7 символов)
- Статус объекта — описание готовности к инспекции

### 3.5 Критерии проверки

Критерии разделены на три части:

#### Часть I: Формальная проверка и структура документа

Проверка титульного листа, метаданных, структуры, терминологии и доступности документа.

#### Часть II: Полнота и содержание требований

Проверка полноты функциональных и нефункциональных требований, границ проекта, требований к интеграции и безопасности.

#### Часть III: Качество и атрибуты требований

Проверка атомарности, проверяемости, прослеживаемости и приоритизации требований.

**Формат таблицы критериев:**

```markdown
| №   | Критерий проверки | Описание | Статус (+/-) | Замечания |
| --- | ----------------- | -------- | :----------: | --------- |
| 1   | **Название**      | Текст    |      +       |           |
| 2   | **Название**      | Текст    |      -       | Текст     |
| 3   | **Название**      | Текст    |     +/-      | Текст     |
```

**Обозначения статусов:**
- `+` — Критерий выполнен полностью
- `-` — Критерий не выполнен
- `+/-` — Критерий выполнен частично

**Правила заполнения колонки "Статус":**
- Колонка "Статус (+/-)" выравнивается **по центру** (`:----------:`)
- Значения статуса: `+`, `-` или `+/-`

**Правила заполнения колонки "Замечания":**
- Если статус `+` (выполнен полностью) — замечания **не указываются** (оставить пустым)
- Если статус `-` или `+/-` — **обязательно** указать конкретное замечание с описанием проблемы

### 3.6 Результаты формальной инспекции

```markdown
**Итоговая оценка:** <Принять | Принять с доработками | Отклонить>

**Рекомендации по исправлению:**
- <замечание 1>
- <замечание 2>
- <замечание 3>
```

**Варианты итоговой оценки:**
- **Принять** — документ соответствует всем критериям, может быть утвержден
- **Принять с доработками** — есть замечания, требующие исправления
- **Отклонить** — документ требует значительной переработки

### 3.7 Подписи

```markdown
**Инспектор**

Подпись: _____________________

ФИО: _____________________

Дата: _____________________
```

**Правила:**
- Обязательные подписи: Инспектор
- Опциональные подписи: Эксперт, Автор документа (при необходимости)
- Формат: подпись, ФИО, дата



## 4. Правила именования файлов протоколов

**Формат:** `<YYYY-MM-DD>_<артефакт>.md`

**Где:**
- `<YYYY-MM-DD>` — дата проведения инспекции (ISO 8601)
- `<артефакт>` — название артефакта (совпадает с именем в `artifacts/`)
- `.md` — расширение Markdown

**Примеры:**
```
✅ validations/1_technical_specification/2025-11-03_technical_specification.md
✅ validations/2_TS_to_SR/2025-12-10_TS_to_SR.md
```



## 5. Расположение файлов

Протоколы инспекции размещаются в папке `validations/<номер>_<артефакт>/`:

```
validations/
├── 1_technical_specification/
│   ├── _inspection_template.md        # Шаблон инспекции
│   └── 2025-11-03_technical_specification.md  # Отчет
├── 2_TS_to_SR/
│   ├── _traceability_analysis_template.md  # Шаблон анализа трассировки
│   └── 2025-12-10_TS_to_SR.md
└── 3_system_requirements/
    ├── _inspection_template.md        # Шаблон инспекции
    └── 2025-12-XX_system_requirements.md
```

**Правила:**
- Шаблон начинается с префикса `_` (служебный файл)
- Отчеты именуются по дате проведения инспекции



## 6. Шаблон протокола

Шаблон протокола формальной инспекции находится в каждой папке артефакта:

```
validations/<номер>_<артефакт>/_inspection_template.md
```

**Использование шаблона:**
1. Скопировать шаблон
2. Переименовать согласно правилам (дата + название артефакта)
3. Заполнить все обязательные разделы
4. Провести инспекцию и заполнить таблицу критериев
5. Указать итоговую оценку и рекомендации



## 7. Процесс проведения инспекции

1. **Подготовка:**
   - Определить версию артефакта для проверки
   - Назначить участников (инспектор, эксперт)
   - Скопировать шаблон и создать файл отчета

2. **Проведение:**
   - Последовательно проверить все критерии
   - Заполнить таблицу: статус и замечания
   - Зафиксировать конкретные проблемы

3. **Завершение:**
   - Сформулировать итоговую оценку
   - Составить список рекомендаций по исправлению
   - Подписать протокол
   - Зафиксировать отчет в Git



## 8. Критерии проверки по умолчанию

Все протоколы используют следующие стандартные критерии проверки:

### Часть I: Формальная проверка и структура документа

1. **Титульный лист и метаданные** — наличие версии, даты, автора, истории изменений
2. **Структура и форматирование** — соответствие шаблону, нумерация разделов
3. **Терминология и грамотность** — наличие глоссария, единообразие терминов
4. **Доступность документа** — размещение в репозитории, актуальность ссылки

### Часть II: Полнота и содержание требований

5. **Цели и границы проекта** — описание целей и границ
6. **Полнота функциональных требований** — описание функций и сценариев
7. **Полнота нефункциональных требований** — производительность, надёжность
8. **Требования к интерфейсам и интеграции** — API, протоколы, форматы
9. **Требования к безопасности** — аутентификация, авторизация, шифрование
10. **Обработка ошибок и исключительных ситуаций** — поведение при сбоях

### Часть III: Качество и атрибуты требований

11. **Атомарность и однозначность** — одно требование = одна вещь
12. **Проверяемость и измеримость** — наличие критериев приёмки
13. **Прослеживаемость до бизнес-целей** — связь с целями проекта
14. **Приоритизация требований** — наличие приоритетов

**Примечание:** Для разных типов артефактов критерии могут адаптироваться с сохранением общей структуры.
