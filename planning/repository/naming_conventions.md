## 1. Цель документа

Этот документ определяет единые правила именования файлов и папок в проекте ITSIMS. Соблюдение этих правил обеспечивает:

- **Консистентность** — все файлы следуют общим соглашениям
- **Читаемость** — названия понятны и информативны
- **Сортировку** — файлы отображаются в логичном порядке
- **Масштабируемость** — система именования работает при росте проекта



## 2. Общие принципы

### 2.1 Язык именования

**Основное правило:** Все файлы и папки именуются **на латинице** 

**Алфавит:**

| Тип файла/объекта | Алфавит  | Примеры                                    |
| ----------------- | -------- | ------------------------------------------ |
| **Все файлы**     | Латиница | `1_technical_specification.md`             |
| **Файлы задач**   | Латиница | `2025-11-29_grechko.md`                    |
| **Артефакты**     | Латиница | `3_system_requirements.md`                 |
| **Трассировка**   | Латиница | `2_TS_to_SR.md`                            |
| **Папки**         | Латиница | `artifacts/`, `planning/`                  |
| **Отчеты**        | Латиница | `2025-11-03_spec_technical_inspection.pdf` |

**Правило:** Кириллица не используется в именах файлов и папок

### 2.2 Стиль именования

**Базовый стиль:** `snake_case` (слова через подчеркивание, строчные буквы)

| Стиль          | Где применяется                                                      | Примеры                               |
| -------------- | -------------------------------------------------------------------- | ------------------------------------- |
| **snake_case** | Все файлы и папки по умолчанию                                       | `system_requirements.md`, `1_sprint/` |
| **kebab-case** | Грамматические исключения (составные слова как единое понятие), даты | `task-tracker/`, `YYYY-MM-DD`         |

**Правило:** По умолчанию используйте `snake_case`. Исключение — составные слова, которые читаются как единый термин (например, "таск-трекер"), пишутся через дефис, а также даты. Пробелы как разделители ЗАПРЕЩЕНЫ.

### 2.3 Формат дат

**Формат:** `YYYY-MM-DD` (ISO 8601, с дефисами)

**Почему этот формат:**
- Международный стандарт ISO 8601
- Естественная сортировка (старые даты → новые даты)
- Однозначность (нет путаницы день/месяц)

### 2.4 Префиксы для сортировки

#### Числовые префиксы (1_, 2_, 3_)

**Назначение:** Устанавливают порядок отображения важных документов

**Где используются:**
- Основные артефакты проекта
- Документы, требующие последовательного чтения
- Главные файлы в каждой категории
- Папки со статусами задач

**Примеры:**
```
artifacts/
├── 1_technical_specification.md # ТЗ
├── 2_TS_to_SR.md                # Трассировка ТЗ ↔ Требования к системе
├── 3_system_requirements.md     # Требования к системе
├── 4_SR_to_SWR.md               # Трассировка Требования к системе ↔ ПО
└── 5_software_requirements.md   # Требования к ПО

task-tracker/
├── 1_sprint/                   # Спринт (первая стадия)
├── 2_in_progress/              # В работе (вторая стадия)
└── 3_done/                     # Выполнено (третья стадия)
```

#### Префикс подчеркивания (_)

**Назначение:** Обозначение служебных файлов и шаблонов в верху файлового менеджера

**Где используется:**
- Шаблоны документов
- Служебные папки
- Вспомогательные ресурсы

**Примеры:**
```
task-tracker/1_sprint/
├── _task_template.md           # Шаблон задачи (служебный)
├── 2025-11-29_grechko.md       # Реальная задача
└── 2025-11-30_koganovsky.md    # Реальная задача

artifacts/
├── _formatting/                # Служебная папка с шаблонами
├── 1_technical_specification.md
├── 2_TS_to_SR.md
└── 3_system_requirements.md
```

**Эффект:** Файлы и папки с `_` отображаются в начале списка, отделяя служебное от рабочего.

### 2.5 Версионирование файлов

Версионирование обеспечивается системой контроля версий. Не добавляйте версии в имена файлов.

❌ Не создавайте версии вручную:

```
❌ artifacts/1_technical_v1.md
❌ artifacts/1_technical_v2.md
❌ artifacts/1_technical_final.md
❌ artifacts/1_technical_final_v2.md
❌ artifacts/1_technical_2025-11-03.md
```

### 2.6 Запрещенные практики

#### ❌ Пробелы в именах

```
❌ system requirements.md
❌ task tracker/
❌ weekly report.md
```

**Почему:** Создают проблемы в командной строке и скриптах.

**Правильно:**
```
✅ system_requirements.md
✅ task-tracker/
✅ weekly_report.md
```

#### ❌ Специальные символы

Запрещены: `! @ # $ % ^ & * ( ) + = { } [ ] | \ ; : ' " < > ? , /`

**Исключения:**
- Дефис (`-`) — разрешен в датах и грамматических исключениях в именах папок
- Подчеркивание (`_`) — основной разделитель
- Точка (`.`) — только перед расширением файла

#### ❌ Транслит

```
❌ otchet.md
❌ spetsifikatsiya.md
❌ sistema_trebovaniya.md
```

**Правильно:**
```
✅ report.md
✅ specification.md
✅ system_requirements.md
```

#### ❌ Кириллица

```
❌ 2025-11-29_Гречко.md
❌ техническое_задание.md
❌ отчет.md
```

**Правильно:**
```
✅ 2025-11-29_grechko.md
✅ 1_technical_specification.md
✅ report.md
```

#### ❌ Дублирование типа в расширении

```
❌ requirements_document.md
❌ script.sh.txt
❌ report.pdf.backup
```

**Правильно:**
```
✅ requirements.md
✅ script.sh
✅ report.pdf
```

#### ❌ Использование заглавных букв без причины

```
❌ Artifacts/
❌ README_FILE.md
❌ System_Requirements.md
```

**Правильно:**
```
✅ artifacts/
✅ README.md              (исключение — стандарт)
✅ system_requirements.md
```



## 3. Правила для конкретных типов файлов

### 3.1 Артефакты проекта

**Расположение:** `artifacts/`

**Формат:** `<номер>_<название>.md`

| Компонент    | Описание                      | Примеры                                   |
| ------------ | ----------------------------- | ----------------------------------------- |
| `<номер>`    | Порядок важности (1, 2, 3...) | `1`, `2`, `3`                             |
| `<название>` | Краткое описание, snake_case  | `technical_specification`, `requirements` |
| `.md`        | Расширение Markdown           | `.md`                                     |

**Примеры:**
```
✅ artifacts/1_technical_specification.md
✅ artifacts/2_TS_to_SR.md
✅ artifacts/3_system_requirements.md
✅ artifacts/4_SR_to_SWR.md
✅ artifacts/5_software_requirements.md
```

### 3.2 Файлы задач

**Расположение:** `task-tracker/{1_sprint|2_in_progress|3_done}/`

**Формат:** `<YYYY-MM-DD>_<фамилия>.md`

| Компонент      | Описание                                                       | Примеры                           |
| -------------- | -------------------------------------------------------------- | --------------------------------- |
| `<YYYY-MM-DD>` | Дата создания задачи (ISO 8601)                                | `2025-11-29`, `2025-12-04`        |
| `<фамилия>`    | Фамилия исполнителя, латиница, с заглавной буквы (capitalized) | `Grechko`, `Baranov`, `Valkovets` |
| `.md`          | Расширение Markdown                                            | `.md`                             |

**Примеры:**
```
✅ task-tracker/1_sprint/2025-11-29_Grechko.md
✅ task-tracker/2_in_progress/2025-12-03_Valkovets.md
✅ task-tracker/3_done/2025-11-22_Baranov.md
```

**Важно:** Дата в имени файла — это дата **создания** задачи, не дедлайн.

### 3.3 Отчеты валидации

**Расположение:** `validations/reports/`

**Формат:** `<YYYY-MM-DD>_<артефакт>.<расширение>`

| Компонент      | Описание                               | Примеры                                   |
| -------------- | -------------------------------------- | ----------------------------------------- |
| `<YYYY-MM-DD>` | Дата проведения валидации              | `2025-11-03`                              |
| `<артефакт>`   | Артефакт, который проверяется          | `technical_specification`, `requirements` |
| `<расширение>` | `.pdf` для финальных, `.tex` для LaTeX | `.pdf`, `.tex`                            |

**Примеры:**
```
✅ validations/reports/2025-11-03_technical.pdf
✅ validations/reports/2025-11-03_technical.tex
✅ validations/reports/2025-12-10_requirements.pdf
```

### 3.4 Еженедельные отчеты

**Расположение:** `task-tracker/weekly_reports/`

**Формат:** `<YYYY-MM-DD>_weekly_report.md`

| Компонент       | Описание                                  | Примеры                    |
| --------------- | ----------------------------------------- | -------------------------- |
| `<YYYY-MM-DD>`  | Дата начала отчетной недели (понедельник) | `2025-11-25`, `2025-12-02` |
| `weekly_report` | Фиксированная часть имени                 | `weekly_report`            |
| `.md`           | Расширение Markdown                       | `.md`                      |

**Примеры:**
```
✅ task-tracker/weekly_reports/2025-11-25_weekly_report.md
✅ task-tracker/weekly_reports/2025-12-02_weekly_report.md
```

### 3.5 Документы планирования

**Расположение:** `planning/{configuration_management|formatting|repository}/`

**Формат:** `<название_документа>.md`

| Компонент              | Описание                               | Примеры                                      |
| ---------------------- | -------------------------------------- | -------------------------------------------- |
| `<название_документа>` | Описательное имя, snake_case, латиница | `repository_structure`, `naming_conventions` |
| `.md`                  | Расширение Markdown                    | `.md`                                        |

**Примеры:**
```
✅ planning/repository/naming_conventions.md
✅ planning/repository/repository_structure.md
✅ planning/formatting/format_selection.md
✅ planning/configuration_management/configuration_management_plan.md
```

**Правило:** Не используйте префиксы для документов планирования, если они не требуют строгого порядка чтения.

### 3.6 Шаблоны

**Расположение:** Любая директория (обычно `_formatting/` подпапка)

**Формат:** `<что_шаблонизируется>_<template|rules>.<расширение>`

| Компонент               | Описание                           | Примеры               |
| ----------------------- | ---------------------------------- | --------------------- |
| `<что_шаблонизируется>` | Имя документа или типа             | `1_technical`, `task` |
| `<template или rules>`  | template = шаблон, rules = правила | `template`, `rules`   |
| `<расширение>`          | Формат шаблона                     | `.md`, `.tex`         |

**Примеры:**
```
✅ artifacts/_formatting/1_technical_template.md
✅ artifacts/_formatting/1_technical_rules.md
✅ task-tracker/1_sprint/_task_template.md
✅ validations/_formatting/spec_formal_inspection_template.tex
```

### 3.7 Служебные скрипты

**Расположение:** `tools/`

**Формат:** `<действие>_<объект>.sh`

| Компонент    | Описание                 | Примеры            |
| ------------ | ------------------------ | ------------------ |
| `<действие>` | Что делает скрипт        | `prepare`          |
| `<объект>`   | С чем работает скрипт    | `for_print`, `all` |
| `.sh`        | Расширение Bash скриптов | `.sh`              |

**Примеры:**
```
✅ tools/printing/prepare_for_print.sh
✅ tools/printing/prepare_all_for_print.sh
```



## 4. Именование папок

### 4.1 Основные директории

**Формат:** `<название>/`

**Стиль:** lowercase, `snake_case` для составных названий

**Примеры:**
```
✅ artifacts/
✅ planning/
✅ validations/
✅ tools/
✅ task-tracker/              # Грамматическое исключение (kebab-case)
```

**Правила:**
- Все папки верхнего уровня — lowercase
- Используйте множественное число, если папка содержит множество однотипных файлов
- По умолчанию `snake_case`, исключение — грамматические составные термины через дефис
- Избегайте сокращений, если они не общеприняты

### 4.2 Служебные подпапки

**Формат:** `_<название>/`

**Назначение:** Папки с шаблонами, правилами, вспомогательными ресурсами

**Примеры:**
```
✅ artifacts/_formatting/
✅ validations/_formatting/
✅ task-tracker/weekly_reports/_itsims.png
```

**Эффект:** Служебные папки отображаются первыми в списке.

### 4.3 Тематические подпапки

**Формат:** `<название>/`

**Примеры:**
```
✅ planning/configuration_management/
✅ planning/formatting/
✅ planning/repository/
✅ task-tracker/1_sprint/
✅ task-tracker/2_in_progress/
✅ task-tracker/3_done/
✅ tools/printing/instructions/
```

**Правила:**
- `snake_case` для составных названий
- Числовые префиксы, если порядок важен (`1_sprint/`, `2_in_progress/`, `3_done/`)
- Логическая группировка (не более 2-3 уровней вложенности)
