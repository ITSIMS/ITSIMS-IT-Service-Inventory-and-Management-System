# Ревизия и актуализация правил форматирования документов

**Создана:** 2025-11-22
**Ответственный:** Когановский
**Нужно сделать до:** 2025-11-28

## Описание

В рамках аудита проекта выявлена необходимость систематизации и актуализации правил оформления документов. Существующие правила разбросаны по разным папкам (`general_rules/`, `artifacts/_formatting/`), требуют ревизии и дополнения, особенно в части трассировки требований между артефактами разных уровней.

**Результат:** Обновленная и систематизированная документация по оформлению, включающая общие и частные правила, а также актуализированные шаблоны для всех типов документов проекта.

---

## Чеклист

### Этап 1: Ревью текущей документации

- [ ] Провести ревью `artifacts/_formatting/1_technical_rules.md`
  - [ ] Проверить актуальность правил для ТЗ
  - [ ] Сверить с текущим состоянием `artifacts/1_technical.md`
  - [ ] Выявить несоответствия между правилами и реальным документом

- [ ] Провести ревью `artifacts/_formatting/1_technical_template.md`
  - [ ] Проверить полноту шаблона
  - [ ] Убедиться в соответствии правилам оформления

### Этап 2: Частные правила для ТЗ

- [ ] Актуализировать `artifacts/_formatting/1_technical_rules.md`
  - [ ] Внести правки на основе ревью
  - [ ] **Особое внимание:** правила трассировки требований
    - [ ] Проверить формат ID требований (с суффиксами документа)
    - [ ] Убедиться в корректности механизма связывания требований
    - [ ] Добавить примеры трассировки (ТЗ → Требования к системе)
  - [ ] Убедиться в единообразии терминов (F1_ST, NF1_ST и т.д.)
  - [ ] Проверить чеклист перед фиксацией документа

- [ ] Актуализировать шаблон `artifacts/_formatting/1_technical_template.md`
  - [ ] Привести в соответствие с обновленными правилами
  - [ ] Проверить полноту всех разделов
  - [ ] Добавить комментарии-подсказки для автора (если нужно)

### Этап 3: Правила и шаблон для требований к системе

- [ ] Создать `artifacts/_formatting/2_system_requirements_rules.md`
  - [ ] Определить специфичные правила для документа требований к системе
  - [ ] **Ключевое внимание:** правила трассировки
    - [ ] Как требования системы покрывают требования ТЗ
    - [ ] Формат ID требований (например, SRS-001, SRS-002)
    - [ ] Таблицы трассировки: формат, обязательные поля
    - [ ] Механизм ссылок между документами (якоря, гиперссылки)
  - [ ] Определить структуру документа требований к системе
  - [ ] Правила формулировки требований (детализация по сравнению с ТЗ)
  - [ ] Чеклист перед фиксацией

- [ ] Создать `artifacts/_formatting/2_system_requirements_template.md`
  - [ ] Разработать полноценный шаблон документа требований к системе
  - [ ] Включить разделы:
    - [ ] Метаданные и лист изменений
    - [ ] Введение и область применения
    - [ ] Функциональные требования
    - [ ] Нефункциональные требования
    - [ ] Ограничения и предположения
    - [ ] **Таблица трассировки требований** (обязательно!)
  - [ ] Добавить примеры заполнения
  - [ ] Обеспечить согласованность с шаблоном ТЗ

### Этап 4: Интеграция и проверка

- [ ] Убедиться в согласованности всех документов
  - [ ] Частные правила соответствуют общепроектным стандартам
  - [ ] Шаблоны соответствуют правилам
  - [ ] Терминология единообразна
  - [ ] Механизм трассировки последователен

- [ ] Провести самопроверку
  - [ ] Попробовать применить правила к существующему `artifacts/1_technical.md`
  - [ ] Проверить, все ли требования имеют корректные ID
  - [ ] Убедиться, что правила можно применить к `artifacts/2_system_requirements.md`

- [ ] Подготовить краткую сводку изменений для команды
  - [ ] Что изменилось в правилах
  - [ ] Новые документы (правила и шаблоны для требований к системе)
  - [ ] Ключевые моменты по трассировке

## Критерии приемки

1. Все существующие правила и шаблоны актуализированы
2. Созданы правила и шаблон для требований к системе
3. Механизм трассировки требований четко описан и применим на практике
4. Все документы согласованы между собой
5. Терминология единообразна во всех документах
6. Правила можно применить к существующим артефактам без противоречий

---

## Примечания

**Особое внимание к трассировке!** Это критически важно для проекта, т.к. мы должны обеспечить прозрачную связь между требованиями разных уровней:
- ТЗ (высокоуровневые требования)
- Требования к системе (детализация)
- Требования к ПО (техническая реализация)

Убедись, что механизм трассировки:
- Прост в применении
- Поддерживается в Markdown
- Позволяет автоматически проверять покрытие
- Обеспечивает двустороннюю связь (от ТЗ к системным требованиям и обратно)
