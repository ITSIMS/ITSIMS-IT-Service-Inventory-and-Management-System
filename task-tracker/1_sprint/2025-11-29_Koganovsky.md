# Создание общих правил оформления документов проекта

**Создана:** 2025-11-29
**Ответственный:** Когановский
**Нужно сделать до:** 2025-11-30

## Описание

Первый этап работы по систематизации правил оформления документов в проекте. Необходимо создать общие правила, которые будут применяться ко всем типам документов проекта (артефакты, планы, отчеты, технические документы).

**Результат:** Набор файлов в папке `project_rules/`, описывающих общие принципы и стандарты оформления документов в репозитории ITSIMS.

---

## Чеклист

### Этап 1: Анализ текущего состояния

- [ ] Изучить существующие файлы в `project_rules/`
  - [ ] `decision_document_format.md`
  - [ ] `unified_document_format.md`
- [ ] Проанализировать общие правила в `.cursor/rules/1_project/`
- [ ] Выявить пробелы в документации общих правил

### Этап 2: Разработка общих правил

- [ ] Создать `project_rules/general_formatting_rules.md`
  - [ ] Основные принципы оформления документов
  - [ ] Использование Markdown как основного формата
  - [ ] Структура заголовков (H1-H6)
  - [ ] Оформление списков (маркированные, нумерованные)
  - [ ] Оформление таблиц
  - [ ] Оформление кода и команд
  - [ ] Использование ссылок и якорей
  - [ ] Правила добавления изображений и диаграмм

- [ ] Создать `project_rules/document_structure_rules.md`
  - [ ] Обязательные элементы документа:
    - [ ] Таблица версионирования (формат, обязательные поля)
    - [ ] Заголовок документа
    - [ ] Оглавление (когда нужно)
    - [ ] Основное содержание
    - [ ] Приложения (при необходимости)
  - [ ] Рекомендуемая последовательность разделов
  - [ ] Правила структурирования больших документов

- [ ] Создать `project_rules/file_naming_rules.md`
  - [ ] Правила именования файлов документов
  - [ ] Правила именования директорий
  - [ ] Использование префиксов и суффиксов
  - [ ] Кодировка имен файлов (латиница vs кириллица)
  - [ ] Правила для вспомогательных файлов (изображения, приложения)

- [ ] Создать `project_rules/versioning_rules.md`
  - [ ] Формат таблицы версионирования
  - [ ] Правила заполнения полей:
    - [ ] Дата изменения
    - [ ] Номер версии (семантическое версионирование)
    - [ ] Автор изменений
    - [ ] Ссылка на задачу
    - [ ] Комментарий к изменению
  - [ ] Когда увеличивать мажорную/минорную версию
  - [ ] Интеграция с Git (связь коммитов и версий документа)

### Этап 3: Дополнительные правила

- [ ] Создать `project_rules/repository_organization_rules.md`
  - [ ] Структура папок проекта
  - [ ] Назначение каждой папки (`artifacts/`, `planning/`, `task-tracker/`, и т.д.)
  - [ ] Правила размещения документов по папкам
  - [ ] Правила работы с вспомогательными папками (`_formatting/`, `resources/`, и т.д.)

- [ ] Актуализировать существующие файлы
  - [ ] Проверить `unified_document_format.md` на соответствие новым правилам
  - [ ] Проверить `decision_document_format.md` на соответствие новым правилам
  - [ ] При необходимости внести правки или переработать

### Этап 4: Валидация и интеграция

- [ ] Проверить согласованность всех созданных правил
  - [ ] Отсутствие противоречий между документами
  - [ ] Единство терминологии
  - [ ] Полнота охвата аспектов оформления

- [ ] Убедиться в применимости правил
  - [ ] Проверить на примере существующих документов проекта
  - [ ] Убедиться, что правила не слишком ограничительны
  - [ ] Убедиться, что правила достаточно конкретны для применения

- [ ] Создать `project_rules/README.md`
  - [ ] Краткое описание всех файлов правил
  - [ ] Навигация по правилам
  - [ ] Рекомендации по использованию правил

## Критерии приемки

1. В папке `project_rules/` созданы все необходимые файлы с общими правилами
2. Все правила написаны на русском языке
3. Правила согласованы между собой и не содержат противоречий
4. Правила применимы к существующим документам проекта
5. Создан README с навигацией по правилам
6. Существующие файлы правил актуализированы

---

## Примечания

**Фокус на общих принципах!** Эта задача направлена на создание базовых правил, применимых ко всем типам документов. Частные правила для конкретных типов документов (ТЗ, требования к системе, отчеты валидации) будут созданы в отдельных задачах.

**Связь с Git:** Убедись, что правила версионирования документов хорошо интегрируются с существующей Git-стратегией проекта (см. `.cursor/rules/2_git/`).

**Преемственность:** При создании правил учитывай уже существующие практики и стандарты проекта, описанные в `.cursor/rules/`.
