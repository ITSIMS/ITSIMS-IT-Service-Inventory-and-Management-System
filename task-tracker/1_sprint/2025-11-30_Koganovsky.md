# Разработка общих правил оформления артефактов и трассировки требований

**Создана:** 2025-11-30
**Ответственный:** Когановский
**Нужно сделать до:** 2025-12-01

## Описание

Второй этап работы по систематизации правил оформления. Необходимо разработать общие правила для всех артефактов проекта (ТЗ, системные требования, требования к ПО) с особым фокусом на механизм трассировки требований между документами разных уровней.

**Результат:** Документ `artifacts/_formatting/general_artifact_rules.md`, описывающий единые стандарты оформления артефактов и механизм трассировки требований.

## Чеклист

### Этап 1: Анализ требований к трассировке

- [ ] Определить уровни требований в проекте
  - [ ] ТЗ (Technical Specification) — высокоуровневые требования заказчика
  - [ ] Требования к системе (System Requirements) — детализация на уровне системы
  - [ ] Требования к ПО (Software Requirements) — техническая реализация
  
- [ ] Определить типы трассировки
  - [ ] Прямая трассировка (от ТЗ → Система → ПО)
  - [ ] Обратная трассировка (от ПО → Система → ТЗ)
  - [ ] Горизонтальная трассировка (между требованиями одного уровня)

- [ ] Изучить существующие практики
  - [ ] Проанализировать текущее состояние `artifacts/1_technical.md`
  - [ ] Проанализировать текущее состояние `artifacts/2_system_requirements.md`
  - [ ] Выявить проблемы текущего подхода

### Этап 2: Разработка системы идентификации требований

- [ ] Разработать схему ID для требований ТЗ
  - [ ] Формат ID (например: `TS-F-001`, `TS-NF-001`)
  - [ ] Префиксы для функциональных требований
  - [ ] Префиксы для нефункциональных требований
  - [ ] Правила нумерации и группировки

- [ ] Разработать схему ID для требований к системе
  - [ ] Формат ID (например: `SRS-F-001`, `SRS-NF-001`)
  - [ ] Связь с ID требований ТЗ
  - [ ] Правила детализации (1 требование ТЗ → N требований системы)

- [ ] Разработать схему ID для требований к ПО
  - [ ] Формат ID (например: `SWRS-F-001`, `SWRS-NF-001`)
  - [ ] Связь с ID требований системы
  - [ ] Правила детализации

- [ ] Обеспечить уникальность и стабильность ID
  - [ ] Правила присвоения новых ID
  - [ ] Что делать при удалении требования
  - [ ] Что делать при разделении требования

### Этап 3: Разработка механизма трассировки

- [ ] Определить способы связывания требований
  - [ ] Использование якорей в Markdown
  - [ ] Использование гиперссылок между документами
  - [ ] Формат ссылок на требования

- [ ] Разработать формат таблиц трассировки
  - [ ] Таблица прямой трассировки (в документе более низкого уровня)
  - [ ] Обязательные поля таблицы
  - [ ] Опциональные поля (статус, комментарий)
  - [ ] Примеры таблиц

- [ ] Определить правила поддержки трассировки
  - [ ] Как обновлять таблицы при добавлении требований
  - [ ] Как обновлять таблицы при изменении требований
  - [ ] Как обновлять таблицы при удалении требований
  - [ ] Проверка полноты покрытия

### Этап 4: Общие правила оформления артефактов

- [ ] Создать `artifacts/_formatting/general_artifact_rules.md`
  - [ ] Общая структура артефакта:
    - [ ] Титульный лист (метаданные)
    - [ ] Таблица версионирования
    - [ ] Оглавление

    - [ ] Язык требований (должно, может, следует)
    - [ ] Атрибуты требований (приоритет, источник, и т.д.)
    - [ ] Критерии проверяемости требований
    - [ ] Запрещенные формулировки

  - [ ] Правила именования артефактов:
    - [ ] Формат имени файла

## Примечания

**Трассировка — критически важна!** Это ключевой аспект задачи. Механизм трассировки должен:
- Быть простым в применении
- Поддерживаться средствами Markdown (якоря, ссылки)
- Обеспечивать двустороннюю прозрачность (видно откуда требование пришло и куда оно ушло)
- Позволять проверять полноту покрытия требований

**Связь уровней требований:**
```
ТЗ (TS-F-001)
  └─> Требования к системе (SRS-F-001, SRS-F-002)
      └─> Требования к ПО (SWRS-F-001, SWRS-F-002, SWRS-F-003)
```

**Преемственность:** Опирайся на общие правила оформления документов, созданные в предыдущей задаче (2025-11-29_Koganovsky.md).

**Стандарты:** При разработке правил можно ориентироваться на IEEE 830 (Software Requirements Specification), но адаптировать под специфику проекта и возможности Markdown.
