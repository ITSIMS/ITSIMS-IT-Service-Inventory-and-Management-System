# Создать шаблон формальной инспекции для системных требований

**Создана:** 2025-12-09
**Ответственный:** Зацепилин
**Нужно сделать до:** 2025-12-12

## Описание

Необходимо создать шаблон для проведения формальной инспекции документа системных требований. Шаблон должен быть адаптирован под специфику требований и обеспечивать систематическую проверку качества документа.

**Контекст:**
- Инспекции технического задания успешно завершены
- Вальковец создаст первую версию системных требований
- Необходим шаблон для инспекции этого документа

**Зависимости:**
- Можно работать параллельно с созданием системных требований
- Учесть опыт инспекций технического задания

**Ожидаемый результат:**
- Шаблон инспекции `inspections/_templates/system_requirements_inspection_template.md`
- Чек-листы для проверки всех аспектов системных требований
- Шаблон готов к использованию для инспекции документа от Вальковца

**Базовые материалы:**
- Шаблоны инспекций для ТЗ (`inspections/_templates/`)
- Опыт проведения инспекций ТЗ
- Правила оформления системных требований (от Когановского)

## Чеклист

### Этап 1: Анализ и подготовка

- [ ] Изучить существующие шаблоны инспекций (для ТЗ)
- [ ] Определить специфику инспекции системных требований
- [ ] Ознакомиться с правилами оформления системных требований (задача Когановского)
- [ ] Определить ключевые аспекты для проверки

### Этап 2: Структура шаблона

- [ ] Создать файл `inspections/_templates/system_requirements_inspection_template.md`
- [ ] Определить разделы шаблона:
  - [ ] Метаинформация (дата, версия документа, инспекторы)
  - [ ] Цель инспекции
  - [ ] Чек-листы проверок
  - [ ] Таблица обнаруженных дефектов
  - [ ] Итоговое заключение
  - [ ] Решение (утвердить/отправить на доработку)

### Этап 3: Чек-листы проверок

- [ ] **Формальная структура:**
  - [ ] Наличие всех обязательных разделов
  - [ ] Корректность титульной информации
  - [ ] Таблица версионирования заполнена
  - [ ] Оглавление актуально
  - [ ] Соответствие шаблону

- [ ] **Качество требований:**
  - [ ] Каждое требование имеет уникальный идентификатор (SR-XXX)
  - [ ] Требования однозначны (одна интерпретация)
  - [ ] Требования измеримы (можно проверить выполнение)
  - [ ] Требования проверяемы (можно протестировать)
  - [ ] Требования полны (вся необходимая информация)
  - [ ] Требования атомарны (одно требование = одна вещь)
  - [ ] Приоритеты требований указаны корректно

- [ ] **Классификация требований:**
  - [ ] Функциональные требования корректно выделены
  - [ ] Нефункциональные требования корректно выделены
  - [ ] Классификация логична и понятна

- [ ] **Трассировка:**
  - [ ] Таблица трассировки TS→SR присутствует
  - [ ] Все требования из ТЗ покрыты
  - [ ] Нет "висящих" системных требований без связи с ТЗ
  - [ ] Идентификаторы в таблице совпадают с идентификаторами в документе

- [ ] **Содержательная полнота:**
  - [ ] Все требования из ТЗ раскрыты на системном уровне
  - [ ] Нет пропущенных аспектов
  - [ ] Детализация достаточна для понимания системы
  - [ ] Введение и контекст понятны

- [ ] **Оформление:**
  - [ ] Соответствие правилам форматирования
  - [ ] Единообразие стиля
  - [ ] Корректность таблиц и списков
  - [ ] Отсутствие опечаток и грамматических ошибок

### Этап 4: Таблица дефектов

- [ ] Создать шаблон таблицы для фиксации дефектов:
  - [ ] Столбец: ID дефекта
  - [ ] Столбец: Раздел/Требование
  - [ ] Столбец: Тип дефекта (критический/значительный/незначительный)
  - [ ] Столбец: Описание дефекта
  - [ ] Столбец: Рекомендация по исправлению
  - [ ] Столбец: Статус (открыт/исправлен/отклонен)

### Этап 5: Итоговое заключение

- [ ] Создать шаблон итогового заключения:
  - [ ] Общая оценка качества документа
  - [ ] Статистика дефектов (по типам)
  - [ ] Основные замечания
  - [ ] Рекомендации
  - [ ] Решение: утвердить / доработать / отклонить

### Этап 6: Финализация

- [ ] Проверить полноту шаблона
- [ ] Убедиться, что шаблон:
  - [ ] Покрывает все аспекты системных требований
  - [ ] Легко использовать
  - [ ] Обеспечивает систематическую проверку
  - [ ] Согласован с правилами проекта
- [ ] Добавить инструкции по использованию шаблона

## Критерии приемки

1. Создан файл `inspections/_templates/system_requirements_inspection_template.md`
2. Шаблон содержит полные чек-листы для проверки всех аспектов системных требований
3. Включена таблица для фиксации дефектов с классификацией по типам
4. Включен раздел итогового заключения с рекомендациями
5. Шаблон учитывает специфику системных требований (трассировка, идентификаторы, классификация)
6. Шаблон согласован с правилами оформления системных требований
7. Шаблон готов к использованию для инспекции документа от Вальковца

## Примечания

**Специфика системных требований:** В отличие от ТЗ, системные требования:
- Имеют формальные идентификаторы (SR-XXX)
- Классифицируются (функциональные/нефункциональные)
- Должны быть трассируемы к ТЗ
- Более детальны и конкретны

**Ключевые аспекты проверки:**
1. **Качество формулировок** — однозначность, измеримость, проверяемость
2. **Трассировка** — полное покрытие требований из ТЗ
3. **Идентификация** — уникальность и корректность идентификаторов
4. **Полнота** — все аспекты системы покрыты

**Опыт инспекций ТЗ:** Используй успешный опыт инспекций технического задания, адаптируя подходы под специфику системных требований.

**Связь с задачами:**
- Шаблон будет использован для инспекции документа от Вальковца
- Отдельный шаблон создается для инспекции таблиц трассировки
