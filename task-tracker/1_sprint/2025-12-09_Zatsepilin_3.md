# Провести формальную инспекцию системных требований

**Создана:** 2025-12-09
**Ответственный:** Зацепилин
**Нужно сделать до:** 2025-12-16

## Описание

Необходимо провести формальную инспекцию документа системных требований, созданного Вальковцем. Инспекция должна выявить все дефекты документа и обеспечить его высокое качество перед утверждением.

**Контекст:**
- Вальковец создаст первую версию системных требований
- Для инспекции будет использован специальный шаблон
- Инспекция аналогична успешным инспекциям технического задания

**Зависимости:**
- Дождаться завершения задачи Вальковца (создание системных требований)
- Дождаться завершения создания шаблона инспекции (собственная задача)

**Ожидаемый результат:**
- Проведена полная формальная инспекция документа
- Создан отчет инспекции `inspections/system_requirements/inspection_v1_0.md`
- Все дефекты зафиксированы и классифицированы
- Даны рекомендации по исправлению
- Принято решение: утвердить / доработать

**Проверяемый документ:**
- `artifacts/2_system_requirements.md` версии 1.0

## Чеклист

### Этап 1: Подготовка к инспекции

- [ ] Дождаться завершения задачи Вальковца (создание системных требований)
- [ ] Дождаться завершения создания шаблона инспекции
- [ ] Изучить правила оформления системных требований (от Когановского)
- [ ] Скопировать шаблон инспекции в `inspections/system_requirements/inspection_v1_0.md`
- [ ] Заполнить метаинформацию (дата, версия документа, инспектор)

### Этап 2: Проверка формальной структуры

- [ ] Проверить наличие всех обязательных разделов:
  - [ ] Титульная информация
  - [ ] Таблица версионирования
  - [ ] Оглавление
  - [ ] Введение
  - [ ] Разделы требований
  - [ ] Таблица трассировки
- [ ] Проверить корректность титульной информации
- [ ] Убедиться, что документ соответствует шаблону
- [ ] Зафиксировать дефекты (если есть)

### Этап 3: Проверка качества требований

Для каждого требования в документе:

- [ ] **Проверка идентификаторов:**
  - [ ] Идентификатор присутствует
  - [ ] Формат корректен (SR-F-XXX, SR-NF-XXX)
  - [ ] Идентификатор уникален
  - [ ] Идентификатор соответствует категории требования

- [ ] **Проверка формулировок:**
  - [ ] Требование однозначно (только одна интерпретация)
  - [ ] Требование измеримо (можно проверить выполнение)
  - [ ] Требование проверяемо (можно протестировать)
  - [ ] Требование полно (вся необходимая информация)
  - [ ] Требование атомарно (описывает одну вещь)

- [ ] **Проверка приоритетов:**
  - [ ] Приоритет указан
  - [ ] Приоритет обоснован
  - [ ] Приоритеты согласованы между требованиями

- [ ] Фиксировать дефекты по каждому проблемному требованию

### Этап 4: Проверка классификации

- [ ] Функциональные требования:
  - [ ] Корректно выделены
  - [ ] Имеют правильные идентификаторы (SR-F-XXX)
  - [ ] Описывают функциональность системы
- [ ] Нефункциональные требования:
  - [ ] Корректно выделены
  - [ ] Имеют правильные идентификаторы (SR-NF-XXX)
  - [ ] Охватывают все категории (производительность, безопасность, надежность и т.д.)
- [ ] Классификация логична и понятна

### Этап 5: Проверка трассировки

- [ ] Таблица трассировки TS→SR присутствует
- [ ] Все требования из ТЗ покрыты системными требованиями
- [ ] Идентификаторы в таблице совпадают с идентификаторами в документе
- [ ] Нет "висящих" системных требований без связи с ТЗ
- [ ] Связи логичны и обоснованы
- [ ] Зафиксировать дефекты трассировки

### Этап 6: Проверка содержательной полноты

- [ ] Все требования из ТЗ раскрыты на системном уровне
- [ ] Детализация достаточна для понимания системы
- [ ] Нет пропущенных аспектов или функциональности
- [ ] Введение и контекст документа понятны
- [ ] Термины и определения корректны
- [ ] Документ читается связно

### Этап 7: Проверка оформления

- [ ] Соответствие правилам форматирования проекта
- [ ] Единообразие стиля изложения
- [ ] Корректность таблиц и списков
- [ ] Отсутствие опечаток
- [ ] Отсутствие грамматических ошибок
- [ ] Ссылки (если есть) работают корректно

### Этап 8: Анализ дефектов

- [ ] Классифицировать все дефекты по типам:
  - [ ] Критические (блокируют утверждение)
  - [ ] Значительные (требуют исправления)
  - [ ] Незначительные (рекомендации)
- [ ] Подсчитать статистику дефектов
- [ ] Выделить наиболее проблемные области
- [ ] Сформулировать рекомендации по исправлению

### Этап 9: Итоговое заключение

- [ ] Написать общую оценку качества документа
- [ ] Указать статистику дефектов
- [ ] Перечислить основные замечания
- [ ] Сформулировать конкретные рекомендации
- [ ] Принять решение:
  - [ ] **Утвердить** — нет критических и значительных дефектов
  - [ ] **Доработать** — есть дефекты, требующие исправления
  - [ ] **Отклонить** — критические проблемы качества

### Этап 10: Финализация отчета

- [ ] Проверить полноту отчета инспекции
- [ ] Убедиться, что все разделы заполнены
- [ ] Проверить корректность таблицы дефектов
- [ ] Убедиться, что итоговое заключение четко и понятно
- [ ] Сохранить отчет: `inspections/system_requirements/inspection_v1_0.md`

## Критерии приемки

1. Проведена полная формальная инспекция документа `artifacts/2_system_requirements.md`
2. Создан отчет инспекции `inspections/system_requirements/inspection_v1_0.md`
3. Проверены все аспекты: структура, требования, трассировка, полнота, оформление
4. Все обнаруженные дефекты зафиксированы в таблице с классификацией
5. Дефекты проанализированы, даны рекомендации по исправлению
6. Принято и зафиксировано решение (утвердить / доработать / отклонить)
7. Отчет соответствует шаблону инспекции

## Примечания

**Систематичность:** Используй шаблон инспекции полностью, проходя по всем чек-листам. Это обеспечит полноту проверки.

**Объективность:** Фиксируй все дефекты, даже незначительные. Классификация по критичности поможет приоритизировать исправления.

**Конструктивность:** Для каждого дефекта давай конкретную рекомендацию по исправлению. Это поможет Вальковцу быстро устранить проблемы.

**Опыт инспекций ТЗ:** Используй опыт успешных инспекций технического задания, применяя те же принципы систематичности и тщательности.

**Ключевые аспекты для проверки:**
1. **Качество требований** — однозначность, измеримость, проверяемость
2. **Трассировка** — полное покрытие требований из ТЗ
3. **Идентификация** — корректность и уникальность идентификаторов
4. **Полнота** — все аспекты системы описаны

**Связь с доработкой:** Если документ потребует доработки, Вальковец получит четкий список дефектов и рекомендаций для исправления. После исправления может потребоваться повторная инспекция.
