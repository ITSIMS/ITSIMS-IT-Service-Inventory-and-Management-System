# Провести формальную инспекцию таблиц трассировки

**Создана:** 2025-12-09
**Ответственный:** Зацепилин

## Описание

Необходимо провести формальную инспекцию таблиц трассировки требований, подготовленных Когановским. Инспекция должна проверить корректность связей между требованиями разных уровней и полноту покрытия.

**Контекст:**
- Когановский приведет в порядок таблицы трассировки TS_to_SR и SR_to_SWR
- Таблицы трассировки критичны для обеспечения качества требований
- Инспекция проводится отдельно от инспекции документа системных требований

**Зависимости:**
- Дождаться завершения задачи Когановского (обновление таблиц трассировки)
- Дождаться завершения создания шаблона инспекции таблиц (собственная задача)

**Ожидаемый результат:**
- Проведена полная формальная инспекция таблиц трассировки
- Создан отчет инспекции `inspections/traceability/inspection_v1_0.md`
- Все дефекты зафиксированы и классифицированы
- Подсчитаны метрики покрытия
- Даны рекомендации по исправлению
- Принято решение: утвердить / доработать

**Проверяемые документы:**
- `artifacts/traceability/TS_to_SR.md`
- `artifacts/traceability/SR_to_SWR.md`

### Этап 2: Инспекция таблицы TS_to_SR

#### 2.1. Проверка формальной структуры

- [ ] Титульная информация присутствует и корректна
- [ ] Таблица версионирования заполнена
- [ ] Структура таблицы корректна (все обязательные столбцы)
- [ ] Заголовки таблицы понятны
- [ ] Форматирование единообразно

#### 2.2. Проверка идентификаторов

- [ ] Все идентификаторы из ТЗ корректны (формат TS-XXX)
- [ ] Все идентификаторы системных требований корректны (формат SR-XXX)
- [ ] Идентификаторы уникальны (нет дублей)
- [ ] Нет опечаток в идентификаторах
- [ ] Идентификаторы соответствуют документам

#### 2.3. Прямая трассировка (TS → SR)

- [ ] Каждое требование из ТЗ присутствует в таблице
- [ ] Каждое требование из ТЗ имеет хотя бы одну связь с SR
- [ ] Связи логичны и обоснованы
- [ ] Нет пропущенных требований из ТЗ

#### 2.4. Обратная трассировка (SR → TS)

- [ ] Каждое системное требование связано с требованием из ТЗ
- [ ] Нет "висящих" системных требований без источника
- [ ] Все системные требования из документа представлены в таблице

#### 2.5. Консистентность

- [ ] Идентификаторы в таблице совпадают с идентификаторами в ТЗ
- [ ] Идентификаторы в таблице совпадают с идентификаторами в системных требованиях
- [ ] Нет ссылок на несуществующие требования
- [ ] Версии документов согласованы

#### 2.6. Метрики покрытия

- [ ] Подсчитать общее количество требований в ТЗ
- [ ] Подсчитать количество требований в ТЗ с связями
- [ ] Рассчитать процент покрытия ТЗ → СТ (цель: 100%)
- [ ] Зафиксировать метрики в отчете

## Примечания

**Критичность трассировки:** Таблицы трассировки — ключевой инструмент управления требованиями. Их качество напрямую влияет на:
- Полноту реализации требований
- Возможность анализа влияния изменений
- Валидацию и верификацию системы
- Соответствие регуляторным требованиям

**Цель инспекции:** Обеспечить 100% покрытие требований на каждом уровне и корректность всех связей.

**Типичные дефекты:**
- **Отсутствие связи** — требование не покрыто
- **Некорректная связь** — требование связано с неправильным элементом
- **Несуществующий ID** — ссылка на несуществующее требование
- **"Висящее" требование** — требование без источника

**Метрики покрытия:**
- **ТЗ → СТ:** Процент требований из ТЗ, имеющих связи с системными требованиями
- **СТ → ТЗ:** Процент системных требований, связанных с требованиями из ТЗ
- **Цель:** 100% в обоих направлениях

**Систематичность:** Используй шаблон полностью, проверяя каждый аспект. Это особенно важно для таблиц трассировки, где легко пропустить дефекты.

**Связь с доработкой:** Если таблицы потребуют доработки, Когановский получит четкий список дефектов. После исправления может потребоваться повторная инспекция.
