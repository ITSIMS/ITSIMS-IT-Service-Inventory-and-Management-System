# Переработать техническое задание на основе диалога с заказчиком

**Создана:** 2025-12-02
**Ответственный:** Вальковец
**Нужно сделать до:** 2025-12-03

## Описание

После создания Когановским правил оформления и шаблона для технического задания необходимо полностью переработать существующее ТЗ (`artifacts/1_technical_specification.md`) в соответствии с требованиями заказчика из диалога.

**Важно:** Создается новая версия документа, таблица версионности продолжается с текущей версии v2.6. Старое ТЗ остается на месте, просто обновляется содержимое файла.

**Базовые материалы для работы:**
- Правила оформления: `artifacts/_formatting/1_technical_rules.md` (от Когановского)
- Шаблон ТЗ: `artifacts/_formatting/1_technical_template.md` (от Когановского)
- Общие правила трассировки: `artifacts/_formatting/general_artifact_rules.md`
- Текущее ТЗ v2.6: `artifacts/1_technical_specification.md`
- Диалог с заказчиком (уточнить требования у Баранова)

---

## Чеклист

### Этап 1: Подготовка

- [ ] Получить от Когановского готовые правила и шаблон
- [ ] Изучить правила оформления ТЗ
- [ ] Проанализировать текущую версию ТЗ (v2.6)
- [ ] Уточнить у Баранова ключевые моменты из диалога с заказчиком

### Этап 2: Переработка структуры

- [ ] Создать новую структуру документа на основе шаблона
- [ ] Определить основные разделы согласно правилам:
  - [ ] Введение с целями и областью применения
  - [ ] Общее описание системы
  - [ ] Функциональные требования
  - [ ] Нефункциональные требования
  - [ ] Ограничения
- [ ] Обеспечить логичную последовательность разделов

### Этап 3: Работа с требованиями

- [ ] Выделить функциональные требования из диалога с заказчиком
- [ ] Присвоить ID функциональным требованиям (формат: `TS-F-XXX`)
- [ ] Выделить нефункциональные требования
- [ ] Присвоить ID нефункциональным требованиям (формат: `TS-NF-XXX`)
- [ ] Создать якоря для каждого требования (для трассировки)
- [ ] Убедиться в уникальности всех ID

### Этап 4: Наполнение содержанием

- [ ] Перенести актуальную информацию из старого ТЗ
- [ ] Дополнить требования на основе диалога с заказчиком
- [ ] Устранить дублирования и противоречия
- [ ] Убедиться в корректности формулировок требований
- [ ] Использовать правильные модальные глаголы (должен, может, следует)

### Этап 5: Финализация

- [ ] Обновить таблицу версионирования (v2.6 → v3.0)
- [ ] Добавить ссылку на эту задачу в таблицу версионности
- [ ] Создать оглавление (если требуется)
- [ ] Проверить работоспособность всех якорей и ссылок
- [ ] Провести самопроверку документа на соответствие правилам

---

## Критерии приемки

1. ТЗ полностью переработан в соответствии с правилами оформления
2. Все требования имеют корректные ID формата `TS-F-XXX` и `TS-NF-XXX`
3. Структура документа соответствует шаблону
4. Требования из диалога с заказчиком полностью отражены
5. Таблица версионирования актуализирована (v3.0)
6. Документ готов к передаче Когановскому на финальную проверку форматирования

## Примечания

**Зависимость:** Работа начинается после получения правил и шаблона от Когановского (задача 2025-12-01_Koganovsky.md, дедлайн 2025-12-02).

**Фокус на содержании:** Основная цель — корректно перенести требования заказчика в структурированный документ с правильной индексацией. Финальное форматирование будет выполнено Когановским позже.

**Трассировка:** Каждое требование должно иметь уникальный ID и якорь для последующей трассировки с требованиями к системе и ПО.

**Связь с задачей Когановского по проверке:** После завершения этой задачи документ будет передан Когановскому для проверки и приведения к правилам оформления (задача 2025-12-03_Koganovsky.md).
