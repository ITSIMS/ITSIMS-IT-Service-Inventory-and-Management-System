# Создать шаблон формальной инспекции для таблиц трассировки

**Создана:** 2025-12-09
**Ответственный:** Зацепилин
**Нужно сделать до:** 2025-12-12

## Описание

Необходимо создать шаблон для проведения формальной инспекции таблиц трассировки требований. Шаблон должен обеспечивать проверку корректности связей между требованиями разных уровней и полноты покрытия.

**Контекст:**
- Когановский приведет в порядок таблицы трассировки TS_to_SR и SR_to_SWR
- Таблицы трассировки — критичный элемент для обеспечения качества требований
- Необходим отдельный шаблон инспекции, фокусирующийся на связях и покрытии

**Зависимости:**
- Можно работать параллельно с обновлением таблиц трассировки
- Учесть специфику проверки трассировки

**Ожидаемый результат:**
- Шаблон инспекции `inspections/_templates/traceability_inspection_template.md`
- Чек-листы для проверки корректности трассировки
- Шаблон готов к использованию для инспекции таблиц от Когановского

**Базовые материалы:**
- Шаблоны инспекций для других документов
- Опыт проведения инспекций
- Понимание структуры таблиц трассировки

## Чеклист

### Этап 1: Анализ и подготовка

- [ ] Изучить существующие шаблоны инспекций
- [ ] Определить специфику инспекции таблиц трассировки
- [ ] Проанализировать требования к трассировке
- [ ] Определить ключевые аспекты для проверки

### Этап 2: Структура шаблона

- [ ] Создать файл `inspections/_templates/traceability_inspection_template.md`
- [ ] Определить разделы шаблона:
  - [ ] Метаинформация (дата, проверяемые таблицы, инспекторы)
  - [ ] Цель инспекции
  - [ ] Список проверяемых таблиц
  - [ ] Чек-листы проверок
  - [ ] Таблица обнаруженных дефектов
  - [ ] Итоговое заключение
  - [ ] Решение

### Этап 3: Чек-листы проверок

- [ ] **Формальная структура таблиц:**
  - [ ] Наличие всех обязательных столбцов
  - [ ] Корректность заголовков
  - [ ] Единообразие формата
  - [ ] Титульная информация заполнена
  - [ ] Таблица версионирования присутствует

- [ ] **Корректность идентификаторов:**
  - [ ] Все идентификаторы исходных требований корректны (TS-XXX)
  - [ ] Все идентификаторы целевых требований корректны (SR-XXX, SWR-XXX)
  - [ ] Идентификаторы уникальны
  - [ ] Идентификаторы соответствуют формату проекта
  - [ ] Нет дублирующихся идентификаторов

- [ ] **Прямая трассировка (TS → SR → SWR):**
  - [ ] Каждое требование из ТЗ имеет связь с системными требованиями
  - [ ] Связи логичны и обоснованы
  - [ ] Нет пропущенных требований
  - [ ] Связи между SR и SWR подготовлены (для будущего)

- [ ] **Обратная трассировка (SWR → SR → TS):**
  - [ ] Каждое системное требование связано с требованием из ТЗ
  - [ ] Нет "висящих" требований без источника
  - [ ] Трассировка позволяет понять происхождение каждого требования

- [ ] **Полнота покрытия:**
  - [ ] Все требования из ТЗ присутствуют в таблице TS_to_SR
  - [ ] Все системные требования присутствуют в таблице SR_to_SWR
  - [ ] Нет потерянных требований
  - [ ] Покрытие 100% на каждом уровне

- [ ] **Консистентность между документами:**
  - [ ] Идентификаторы в таблицах совпадают с идентификаторами в документах
  - [ ] Нет расхождений в формулировках
  - [ ] Нет ссылок на несуществующие требования
  - [ ] Версии документов согласованы

- [ ] **Логика связей:**
  - [ ] Связи "один-ко-многим" корректны
  - [ ] Связи "многие-к-одному" корректны
  - [ ] Связи логически обоснованы
  - [ ] Нет циклических зависимостей

- [ ] **Оформление:**
  - [ ] Таблицы читаемы
  - [ ] Форматирование единообразно
  - [ ] Отсутствие опечаток
  - [ ] Соответствие правилам проекта

### Этап 4: Таблица дефектов

  - [ ] Столбец: Описание дефекта
  - [ ] Столбец: Рекомендация по исправлению
  - [ ] Столбец: Статус

### Этап 5: Метрики покрытия

- [ ] Добавить раздел для подсчета метрик:
  - [ ] Общее количество требований в ТЗ
  - [ ] Количество требований в ТЗ с связями
  - [ ] Процент покрытия ТЗ → СТ

## Примечания

**Специфика инспекции трассировки:** Проверка таблиц трассировки отличается от проверки обычных документов:
- Фокус на корректности связей, а не на содержании
- Важна полнота покрытия (100%)
- Критична консистентность идентификаторов
- Необходима проверка в обе стороны (прямая и обратная трассировка)

**Типичные дефекты трассировки:**
- Отсутствие связи (требование не покрыто)
- Некорректная связь (требование связано с неправильным элементом)
- Несуществующий идентификатор (ссылка на несуществующее требование)
- Дублирование связей
- "Висящие" требования (без источника)

**Важность метрик:** Метрики покрытия — ключевой показатель качества трассировки. 100% покрытие должно быть целью.

**Связь с задачами:**
- Шаблон будет использован для инспекции таблиц от Когановского
- Инспекция таблиц проводится отдельно от инспекции документа системных требований
