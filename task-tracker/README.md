## Введение

Таск-трекер — это система управления задачами команды проекта **ITSIMS** (IT Service Inventory and Management System). Система организована на основе файловой структуры в формате Markdown.

---

## Зачем нужен таск-трекер

Файловый таск-трекер обеспечивает:

- **Полную прозрачность работы команды** — все задачи доступны в репозитории
- **Версионирование задач через Git** — история изменений сохраняется автоматически
- **Отсутствие зависимости от внешних сервисов** — работает офлайн, без подписок
- **Простоту интеграции с документацией** — единый формат Markdown для всего проекта

## Жизненный цикл и правила пользования таск-трекером

### Структура каталогов

```
task-tracker/
├── README.md                       # Данное руководство
├── 1_sprint/                       # Задачи текущего спринта
│   ├── _task_template.md           # Шаблон для создания задач
│   ├── <задача 1>.md               # Пример задачи
│   ├── <задача 2>.md               # Пример задачи
│   └── ...                         # Прочие задачи
├── 2_in_progress/                  # Задачи в работе
│   ├── <задача 3>.md               # Пример задачи в работе
│   └── ...                         # Прочие задачи в работе
├── 3_done/                         # Завершённые задачи
│   ├── <задача 4>.md               # Пример завершённой задачи
│   └── ...                         # Прочие завершённые задачи
└── weekly_reports/                 # Еженедельные отчёты
    ├── _template_weekly_report.md  # Шаблон отчёта
    ├── _itsims.png                 # Логотип команды
    ├── <20251129_weekly_report.md> # Пример еженедельного отчёта
    ├── <20251206_weekly_report.md> # Пример еженедельного отчёта
    └──  ...                        # Прочие отчёты
```

| Каталог           | Назначение                                         |
| ----------------- | -------------------------------------------------- |
| `1_sprint/`       | Задачи, запланированные на текущий спринт (бэклог) |
| `2_in_progress/`  | Задачи, которые находятся в активной работе        |
| `3_done/`         | Архив завершённых задач                            |
| `weekly_reports/` | Еженедельные отчёты о проделанной работе           |

### Жизненный цикл задачи

Каждая задача проходит три стадии:

```
┌─────────────┐      ┌──────────────────┐      ┌─────────────┐
│  1_sprint/  │ ───▶ │  2_in_progress/  │ ───▶ │   3_done/   │
│  (Бэклог)   │      │    (В работе)    │      │  (Готово)   │
└─────────────┘      └──────────────────┘      └─────────────┘
```

1. **Создание задачи** → файл помещается в `1_sprint/`
2. **Начало работы** → файл перемещается в `2_in_progress/`
3. **Завершение** → файл перемещается в `3_done/`

> **Важно:** Перемещение файлов выполняется через Git (`git mv`), чтобы сохранить историю изменений.

### Интеграция с Git

При работе с таск-трекером используйте область `tasks` в сообщениях коммитов:

| Действие             | Сообщение коммита                               |
| -------------------- | ----------------------------------------------- |
| Создание задачи      | `feat(tasks): добавить задачу для Когановского` |
| Изменение задачи     | `fix(tasks): исправить срок выполнения задачи`  |
| Перемещение в работу | `chore(tasks): взять задачу в работу`           |
| Завершение задачи    | `chore(tasks): завершить задачу Баранова`       |
| Создание отчёта      | `feat(tasks): добавить еженедельный отчёт`      |

Перемещение задач между каталогами:

```bash
# Взять задачу в работу
git mv task-tracker/1_sprint/2025-11-28_Baranov.md task-tracker/2_in_progress/
git commit -m "chore(tasks): взять задачу в работу"

# Завершить задачу
git mv task-tracker/2_in_progress/2025-11-28_Baranov.md task-tracker/3_done/
git commit -m "chore(tasks): завершить задачу Баранова"
```

### Принципы эффективной работы

**Одна задача — один исполнитель.** Каждая задача назначается конкретному участнику команды. Это обеспечивает чёткую ответственность, прозрачность загрузки и простоту отслеживания.

**Атомарность задач.** Задача должна быть достаточно конкретной, чтобы её можно было выполнить за один спринт. Если задача слишком большая — разбейте её на несколько.

**Измеримость результата.** Результат выполнения задачи должен быть проверяемым: создан документ, исправлена ошибка, обновлена конфигурация.

**Своевременное обновление статуса.** Начали работу — переместите в `2_in_progress/`. Завершили — переместите в `3_done/`. Не забывайте коммитить изменения.

## Правила оформления задач

### Именование файлов

Формат имени файла:

```
<ГГГГ-ММ-ДД>_<Фамилия-латиницей>.md
```

| Компонент            | Описание                                        | Пример       |
| -------------------- | ----------------------------------------------- | ------------ |
| `ГГГГ-ММ-ДД`         | Дата создания задачи в ISO-формате              | `2025-11-28` |
| `Фамилия-латиницей`  | Фамилия исполнителя латинскими буквами (транслит) | `Baranov`    |

Примеры корректных имён:

```
2025-11-28_Baranov.md
2025-11-28_Zatsepilin.md
2025-12-01_Koganovsky.md
```

#### Индексация при нескольких задачах

Если у одного исполнителя несколько задач с одинаковой датой создания, к имени файла добавляется числовой индекс:

**Формат:** `<ГГГГ-ММ-ДД>_<Фамилия>_<индекс>.md`

Примеры:

```
2025-12-09_Koganovsky_1.md    # первая задача Когановского за 09.12
2025-12-09_Koganovsky_2.md    # вторая задача Когановского за 09.12
2025-12-09_Zatsepilin_1.md
2025-12-09_Zatsepilin_2.md
```

**Правила:**
- Индекс начинается с `_1`
- Индексы присваиваются в порядке создания задач
- Соответствующие ветки Git также индексируются: `task/2025-12-09_koganovsky_1`

Примеры некорректных имён:

```
❌ task_baranov.md         # Нет даты
❌ 28-11-2025_Baranov.md   # Неверный формат даты
❌ 2025-11-28_Ivan.md      # Имя вместо фамилии
❌ 2025-11-28_Баранов.md   # Кириллица
```

### Участники команды

При назначении задачи используйте фамилии на русском языке в поле `assignee`, а в имени файла — латинскими буквами:

| Фамилия (RU) | Фамилия (EN) | Роль           | Область ответственности     |
| ------------ | ------------ | -------------- | --------------------------- |
| Баранов      | Baranov      | Тимлид         | Координация, ревью          |
| Зацепилин    | Zatsepilin   | Верификация    | Проверка качества           |
| Гречко       | Grechko      | Хранилище      | Git, структура файлов       |
| Когановский  | Koganovsky   | Форматирование | Оформление документов       |
| Вальковец    | Valkovets    | Документация   | Написание документов        |
| Носов        | Nosov        | План КУ        | Конфигурационное управление |
| Ловков       | Lovkov       | Поддержка      | Техническая поддержка       |

### Создание новой задачи

**Шаг 1.** Скопируйте файл [_task_template.md](1_sprint/_task_template.md) и переименуйте его согласно правилам именования.

**Шаг 2.** Заполните YAML-заголовок файла:

```yaml
---
created_date: "2025-11-28"
due_date: "2025-12-05"
assignee: "Баранов"
title: "Разработать API для модуля авторизации"
description: "Спроектировать и описать REST API эндпоинты для модуля авторизации пользователей"
---
```

**Шаг 3.** Оформите тело задачи:

```markdown
# Разработать API для модуля авторизации

**Создана:** 2025-11-28
**Ответственный:** Баранов
**Нужно сделать до:** 2025-12-05

## Описание
Спроектировать и описать REST API эндпоинты для модуля авторизации пользователей.

---
*Дополнительные заметки, чек-листы и комментарии*
```

**Шаг 4.** Закоммитьте задачу:

```bash
git add task-tracker/1_sprint/2025-11-28_Baranov.md
git commit -m "feat(tasks): добавить задачу для Баранова"
```

### Заголовок задачи

| Правило                           | Хорошо ✅                          | Плохо ❌                   |
| --------------------------------- | --------------------------------- | ------------------------- |
| Начинается с глагола в инфинитиве | «Разработать API модуля»          | «API модуля»              |
| Максимум 60 символов              | «Обновить раздел требований»      | «Обновить и дополнить...» |
| Без точки в конце                 | «Исправить ошибку в спецификации» | «Исправить ошибку.»       |
| Конкретная формулировка           | «Добавить таблицу пользователей»  | «Доработать документ»     |

### Описание задачи

Описание должно отвечать на три вопроса:

1. **Что** именно нужно сделать?
2. **Какой результат** ожидается?
3. **Где** находятся связанные материалы? (при необходимости)

**Пример хорошего описания:**

> Необходимо разработать раздел «Требования к интерфейсу» в техническом задании. Результатом должен быть заполненный раздел 4.3 документа `deliverables/spec_technical.md` с описанием не менее 10 экранов системы. За основу взять макеты из папки `design/mockups/`.

### Чеклист (опционально)

Для сложных задач добавляйте чеклист в тело документа:

```markdown
## Чеклист
- [ ] Подготовить черновик
- [ ] Согласовать с тимлидом
- [ ] Внести правки после ревью
- [ ] Финализировать документ
- [ ] Обновить связанные разделы
```

## Создание еженедельных отчётов

### Назначение отчётов

Еженедельные отчёты фиксируют прогресс команды и служат основой для планирования следующего спринта.

### Формат имени файла отчёта

```
<ГГГГММДД>_weekly_report.md
```

Где `ГГГГММДД` — дата последнего дня отчётной недели (обычно пятница).

**Пример:** `20251129_weekly_report.md`

### Структура отчёта

1. **Сводка по статусам** — количественные показатели
2. **Детализация задач** — таблица со всеми задачами недели
3. **Краткое резюме** — тезисное описание проделанной работы

### Создание отчёта

1. Скопируйте шаблон [_template_weekly_report.md](weekly_reports/_template_weekly_report.md)
2. Переименуйте файл согласно формату
3. Заполните все секции на основе данных из `3_done/` и `2_in_progress/`
4. Закоммитьте с сообщением:

```bash
git commit -m "feat(tasks): добавить еженедельный отчёт за 21-28 ноября"
```
