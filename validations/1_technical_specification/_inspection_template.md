## Критерии проверки

### Часть I: Формальная проверка и структура документа

| №   | Критерий проверки               | Описание                                                                                                                               | Статус (+/-) | Замечания |
| --- | ------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- | :----------: | --------- |
| 1   | **Титульный лист и метаданные** | Присутствуют название проекта, автор, дата, версия документа и история изменений.                                                      |              |           |
| 2   | **Структура и форматирование**  | Документ соответствует корпоративному шаблону. Разделы и пункты пронумерованы корректно. Все рисунки и таблицы имеют подписи и номера. |              |           |
| 3   | **Терминология и грамотность**  | Присутствует глоссарий. Термины используются единообразно. Отсутствуют орфографические и грамматические ошибки.                        |              |           |
| 4   | **Доступность документа**       | Документ размещён в общем репозитории, ссылка на него актуальна и доступна для всех участников.                                        |              |           |

### Часть II: Полнота и содержание требований

| №   | Критерий проверки                              | Описание                                                                                                               | Статус (+/-) | Замечания |
| --- | ---------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- | :----------: | --------- |
| 5   | **Цели и границы проекта**                     | Четко описаны бизнес-цели, назначение системы, а также то, что входит и НЕ входит в рамки проекта.                     |              |           |
| 6   | **Полнота функциональных требований**          | Описаны все ключевые функции, пользовательские сценарии и/или пользовательские истории.                                |              |           |
| 7   | **Полнота нефункциональных требований**        | Заданы требования к производительности, надёжности, масштабируемости, удобству использования и т.д.                    |              |           |
| 8   | **Требования к интерфейсам и интеграции**      | Определены все внешние и внутренние API, протоколы взаимодействия и форматы данных для интеграции с другими системами. |              |           |
| 9   | **Требования к безопасности**                  | Описаны требования по аутентификации, авторизации, шифрованию данных и защите от известных уязвимостей.                |              |           |
| 10  | **Обработка ошибок и исключительных ситуаций** | Определено поведение системы в случае сбоев, неверного ввода данных и других нештатных ситуаций.                       |              |           |

### Часть III: Качество и атрибуты требований

| №   | Критерий проверки                    | Описание                                                                                                                                                          | Статус (+/-) | Замечания |
| --- | ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- | :----------: | --------- |
| 11  | **Атомарность и однозначность**      | Каждое требование описывает одну и только одну вещь, не допуская двойного толкования или расплывчатых формулировок.                                               |              |           |
| 12  | **Проверяемость и измеримость**      | Для каждого требования можно сформулировать чёткий критерий приёмки и разработать тест-кейс. Формулировки измеримы (например, "время отклика не более 2 секунд"). |              |           |
| 13  | **Прослеживаемость до бизнес-целей** | Каждое требование может быть отслежено до бизнес-цели или запроса от заинтересованной стороны. Понятно, ЗАЧЕМ оно нужно.                                          |              |           |
| 14  | **Приоритизация требований**         | Требованиям присвоены приоритеты, что помогает планировать итерации разработки.                                                                                   |              |           |
