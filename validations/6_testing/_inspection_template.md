## Критерии проверки

### Часть I: Формальная структура и оформление документа


| №   | Критерий проверки               | Описание                                                                                                                                      | Статус (+/-) | Замечания |
| --- | ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- | ------------ | --------- |
| 1   | **Титульный лист и метаданные** | Присутствуют название проекта, тип тестового артефакта, автор, дата, версия документа и история изменений.                                    |              |           |
| 2   | **Структура документа**         | Документ соответствует выбранному типу артефакта (план, набор тестов, отчёт). Разделы и пункты пронумерованы корректно, оглавление актуально. |              |           |
| 3   | **Терминология и грамотность**  | Термины используются единообразно и согласованы с ТЗ и другими артефактами. Отсутствуют орфографические и грамматические ошибки.              |              |           |
| 4   | **Доступность документа**       | Документ размещён в общем репозитории, ссылка на него актуальна и доступна для участников команды.                                            |              |           |


### Часть II: Содержательная полнота тестового артефакта


| №   | Критерий проверки                                | Описание                                                                                                                              | Статус (+/-) | Замечания |
| --- | ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------- | ------------ | --------- |
| 5   | **Связь с требованиями**                         | В документе явно указано, какие требования (TS/SR/SWR) покрываются данным тестовым артефактом.                                        |              |           |
| 6   | **Полнота покрытия критичных требований**        | Приоритетные требования (категория Must have) имеют тестовые сценарии или иные формы проверки.                                        |              |           |
| 7   | **Ясность критериев прохождения**                | Для тестов или групп тестов определены чёткие критерии успешного/неуспешного выполнения.                                              |              |           |
| 8   | **Согласованность статусов тестов (если отчёт)** | В тестовом отчёте статусы тестов (пройдено/не пройдено/заблокировано) интерпретируются однозначно и согласованы с описанием дефектов. |              |           |


### Часть III: Качество описания и пригодность к использованию


| №   | Критерий проверки                  | Описание                                                                                                                                    | Статус (+/-) | Замечания |
| --- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- | ------------ | --------- |
| 9   | **Понятность шагов и сценариев**   | Шаги тестов описаны так, чтобы другой участник команды мог воспроизвести проверку без дополнительных устных пояснений.                      |              |           |
| 10  | **Однозначность формулировок**     | Описания тестов и результатов не допускают двойного толкования; нет оценочных формулировок без конкретных критериев.                        |              |           |
| 11  | **Актуальность и согласованность** | Тестовый артефакт согласован с актуальной версией требований и архитектурных решений; нет явных противоречий с другими документами проекта. |              |           |
