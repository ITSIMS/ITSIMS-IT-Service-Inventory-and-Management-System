## Назначение директории

Директория `validations/` содержит **отчеты проверок и формальных инспекций** документов проекта.

**Что здесь хранится:**
- Отчеты формальных инспекций артефактов
- Результаты валидаций документов
- Шаблоны для проведения инспекций

**Что здесь НЕ хранится:**
- Черновики отчетов
- Внутренние заметки о проблемах
- Рабочие документы



## Структура директории

```
validations/
├── README.md                   # Данный документ
├── reports/                    # Отчеты инспекций
│   ├── 2025-11-03_spec_technical_formal_inspection_report_v1.0.pdf
│   └── 2025-11-03_spec_technical_formal_inspection_report_v1.0.tex
└── _formatting/                # Шаблоны для инспекций
    ├── spec_formal_inspection_template.pdf
    └── spec_formal_inspection_template.tex
```



## Правила работы с отчетами

### Именование отчетов

Формат имени файла:

```
<ГГГГ-ММ-ДД>_<тип>_<название>_v<версия>.<расширение>
```

**Компоненты:**
- `ГГГГ-ММ-ДД` — дата проведения инспекции
- `<тип>` — тип документа (например, `spec`)
- `<название>` — краткое название инспекции (например, `technical_formal_inspection_report`)
- `v<версия>` — номер версии отчета (например, `v1.0`)
- `<расширение>` — формат файла (`.pdf`, `.tex`, `.md`)

**Примеры:**
```
✅ 2025-11-03_spec_technical_formal_inspection_report_v1.0.pdf
✅ 2025-12-04_system_requirements_inspection_v1.0.tex
✅ 2025-12-10_architecture_review_report_v2.0.md

❌ inspection_report.pdf                    # Нет даты и структуры
❌ 2025-11-03_report.pdf                    # Не указан тип
❌ technical_inspection_v1.pdf              # Нет даты
```

### Форматы отчетов

Отчеты могут быть в одном из следующих форматов:

| Формат  | Когда использовать                          |
| ------- | ------------------------------------------- |
| **PDF** | Финальная версия для сдачи и архива         |
| **TeX** | Исходник LaTeX документа для редактирования |
| **MD**  | Отчеты в формате Markdown для простоты      |

**Рекомендация:** Хранить и PDF, и исходник (TeX или MD) для возможности правок.

### Шаблоны инспекций

Шаблоны для проведения инспекций хранятся в `_formatting/`:

```
_formatting/
├── spec_formal_inspection_template.pdf  # Шаблон в PDF
└── spec_formal_inspection_template.tex  # Исходник шаблона
```

Шаблоны помогают стандартизировать формат отчетов и обеспечивают единообразие.



## Создание отчета формальной инспекции

**Шаг 1.** Проведите инспекцию согласно шаблону из `_formatting/`

**Шаг 2.** Подготовьте отчет в выбранном формате (LaTeX или Markdown)

**Шаг 3.** Сохраните файл в `validations/reports/` с правильным именем:
```
validations/reports/2025-12-04_system_requirements_inspection_v1.0.tex
```

**Шаг 4.** Если используете LaTeX, создайте PDF версию:
```bash
cd validations/reports/
pdflatex 2025-12-04_system_requirements_inspection_v1.0.tex
```

**Шаг 5.** Закоммитьте оба файла:
```bash
git add validations/reports/2025-12-04_system_requirements_inspection_v1.0.*
git commit -m "feat(validation): добавить отчет инспекции системных требований"
```



## Типичные сценарии

### Инспекция нового артефакта

**Ситуация:** Завершили работу над техническим заданием, нужно провести формальную инспекцию.

**Действия:**
1. Используй шаблон `_formatting/spec_formal_inspection_template.tex`
2. Заполни все разделы отчета
3. Сохрани как `2025-12-04_spec_technical_formal_inspection_report_v1.0.tex`
4. Создай PDF версию
5. Закоммить оба файла с типом `feat(validation)`

### Повторная инспекция после правок

**Ситуация:** После первой инспекции внесли правки, нужна повторная проверка.

**Действия:**
1. Создай новый отчет с увеличенной версией: `v2.0`
2. Укажи в отчете ссылку на предыдущую версию
3. Закоммить с типом `feat(validation)` или `fix(validation)` в зависимости от характера изменений
